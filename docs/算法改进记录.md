## 当前论文主线

- 研究对象：面向 MANET/UAV-inspired 场景的动态 TDMA/STDMA 时隙调度。
- 主方法：OMNeT++ 在线仿真 + Python PPO/LSTM 调度器，核心学习方法在实验表中记为 `B`，同时保留 `ppo_baseline` 作为弱化 RL 对照。
- 当前主表候选：v3 正式 MANET 对比实验，目录 `logs/benchmark_manet_formal_compare_v3_20260510_151910`。
- v4 结论：饥饿惩罚能明显压低 sparse 场景 starvation，但 pedestrian 场景吞吐/PDR 回退，因此 v4 不直接替代 v3；更适合作为 starvation/fairness trade-off 或 reward 修复消融。
- 论文 baseline 口径：`*_inspired` 和 `*_like` 只能写作“论文启发式近似/本地复现”，不能宣称完整复现原论文协议。

### `B_masked` two-hop mask pilot

- 代码入口：`B_masked`，在 PPO 采样与更新中屏蔽上一帧一跳/两跳邻居占用 slot；安全 slot 使用 `action_init_prob=0.75` 初始化。
- smoke：`logs/benchmark_b_masked_smoke_20260514_try1`，5 updates 跑通，确认 mask、PPO 更新和 summary 管线可用。
- pilot：`logs/benchmark_b_masked_pilot_20260514_try1`，`B` vs 硬 mask 版，UAV sparse，seed 1，`target_updates=100`。硬 mask 版 goodput/PDR `0.1695/0.0835`，低于同预算 `B` 的 `0.1807/0.0887`。
- pilot：`logs/benchmark_b_masked_init075_pilot_20260514_try1`，加入 `action_init_prob=0.75` 后，UAV sparse goodput/PDR 提升至 `0.1971/0.0972`，queue_slope `4.4299`，优于同预算 `B`；但 starvation 升至 `0.8750`。
- pilot：`logs/benchmark_b_masked_starvation_pilot_20260514_try1`，再加 `starvation_penalty_coef=0.1` 后 goodput/PDR `0.2000/0.0988`，但 starvation 仍 `0.8750`，queue_slope 回升至 `6.7547`。
- 结论：two-hop mask + safe-slot aggressive initialization 是有希望的吞吐方向，但不能单独解决公平/饥饿；下一步应把“安全候选 + per-node service debt / deficit round-robin”结合，而不是仅靠原 starvation penalty。

### `B_masked_debt` service debt pilot

- 代码入口：`B_masked_debt`，继承 `B_masked` 的 `action_mask=twohop` 与 `action_init_prob=0.75`，新增 `service_debt_threshold`、`service_debt_action_boost`、`service_debt_reward_coef`、`service_debt_max_frames`。
- 实现修正：还债成功 reward 使用动作发出时记录的 service debt，而不是下一帧观测重新计算的 debt；否则队列被成功清空时可能漏掉还债奖励。
- smoke：`logs/benchmark_b_masked_debt_smoke_20260514_try1`，UAV sparse，seed 1，`target_updates=5`，PPO 到达 target，日志确认 service debt 参数透传。
- 默认 pilot：`logs/benchmark_b_masked_debt_pilot_fix_default_20260514_try1`，参数 `threshold=5, action_boost=0.25, reward_coef=0.5, max_frames=20`。UAV sparse seed 1，`target_updates=100`，goodput/PDR `0.2171/0.1071`，queue_slope `8.1446`，starvation `0.5000`。
- 调参 pilot：`logs/benchmark_b_masked_debt_pilot_fix_tuned_20260514_try1`，参数 `action_boost=0.15, reward_coef=0.75`。goodput/PDR `0.2089/0.1029`，starvation `0.1250`，但 queue_slope `10.4028`，略高于旧 B 参考 `9.8457`。
- 结论：服务债务方向有效，但默认版只贴近 starvation 目标线，调参版存在 queue_slope 代价；本轮不进入小正式，也不替代当前论文主线。下一步若继续，应做“债务优先但总申请预算受控”的动作侧机制，而不是简单提高所有安全槽申请概率。

### `B_masked_debt_budget` controlled service debt

- 代码入口：`B_masked_debt_budget`，在 `B_masked_debt` 基础上增加 `service_debt_request_budget` 与 `service_debt_budget_boost`，按节点对有效安全槽的期望申请总量做上限缩放。
- 默认参数：`action_mask=twohop`，`action_init_prob=0.75`，`service_debt_threshold=5`，`service_debt_action_boost=0.15`，`service_debt_reward_coef=0.75`，`service_debt_max_frames=20`，`service_debt_request_budget=5.0`，`service_debt_budget_boost=3.0`。
- smoke：`logs/benchmark_b_masked_debt_budget_smoke_20260514_try1`，UAV sparse，seed 1，`target_updates=5`，确认 budget 参数透传与 PPO 更新可用。
- pilot：`logs/benchmark_b_masked_debt_budget_pilot_boost3_20260514_try1`，UAV sparse seed 1，`target_updates=100`。goodput/PDR `0.2057/0.1013`，queue_slope `8.5985`，starvation `0.2500`，相比同预算 B 和旧 B 参考均满足平衡目标。
- 小正式：`logs/benchmark_b_masked_debt_budget_small_formal_20260514_try1`，两个 MANET 主场景 × `B B_masked_debt_budget greedy_stdma_2hop` × seeds `1 2 3`，`target_updates=400`。完整性：18/18 `run_status.tsv`，12/12 PPO 日志达到 `target_updates reached: 400`。
- 小正式结论：UAV sparse 中 `B_masked_debt_budget` 相比 `B` 提升 goodput/PDR `0.1781/0.0874 -> 0.1994/0.0980`，queue_slope `9.7940 -> 8.9667`，starvation `0.5417 -> 0.3333`；但 MAC效率与 Jain尾100 下降，且仍低于 `greedy_stdma_2hop` 的吞吐/PDR。
- 小正式结论：pedestrian 中 `B_masked_debt_budget` 相比 `B` goodput/PDR 下降 `0.2332/0.1145 -> 0.2091/0.1026`，但 starvation 改善 `0.7917 -> 0.5833`，queue_slope 略升 `16.7748 -> 17.4514`。
- 论文口径：可作为 UAV sparse fairness/stability 消融或附录候选，不宜替代 v3 主方法；适合写成“two-hop safety prior + controlled service-debt scheduling improves sparse topology stability at the cost of pedestrian throughput and per-slot efficiency”。

### `B_masked_debt_adaptive` density-aware debt budget

- 代码入口：`B_masked_debt_adaptive`，继承 `B_masked_debt_budget` 的 two-hop mask、`action_init_prob=0.75`、service debt 与 request budget，并新增密度自适应开关和动态预算控制。
- C++/JSON 扩展：`writeRlFeatures()` 新增 `network.active_nodes`、`network.active_edges`，Python `rl_receiver.NodeObservation` 兼容解析；密度因子由 `active_edges / (active_nodes*(active_nodes-1)/2)` 与 safe-slot ratio 共同决定。
- 动作侧：稀疏场景保留 two-hop mask 与安全槽先验；密集场景关闭 two-hop mask，并把 actor 概率向 `0.5` 回退，避免严格安全 mask 在高密度拓扑中过度限制候选槽。
- 还债侧修正：service debt 的 action boost 与成功 reward 不再随密度因子衰减，只让密度开关控制 mask/安全先验；否则密集或半密集场景会削弱长期未服务节点的“还债”能力。
- 动态预算：默认 `service_debt_budget_min_scale=0.90`，根据近期申请成功率和队列增长在 `[0.90, 1.10]` 内缩放 request budget；pilot 中多数窗口仍处于保守下限，后续可继续调 `success_target`。
- smoke：`logs/benchmark_b_masked_debt_adaptive_smoke_20260514_try1`，UAV sparse，seed 1，`target_updates=5`，PPO 到达 target，日志确认 `density_adaptive=True`、`dynamic_budget=True`，并输出 `density/edge/safe/budget_scale`。
- pilot try2：`logs/benchmark_b_masked_debt_adaptive_pilot_20260514_try2`，两个 MANET 主场景 × `B B_masked_debt_budget B_masked_debt_adaptive` × seed 1，`target_updates=100`。初版 adaptive 在 UAV sparse goodput/PDR `0.1847/0.0908`，queue_slope `8.8116`，但 starvation `0.5000`；pedestrian starvation `0.8750`，说明过度节流和 debt 衰减不合适。
- pilot try3：`logs/benchmark_b_masked_debt_adaptive_pilot_20260514_try3`，仅重跑修正版 adaptive。pedestrian：goodput/PDR `0.2168/0.1069`，queue_slope `16.1716`，starvation `0.6250`；UAV sparse：goodput/PDR `0.1868/0.0920`，queue_slope `9.2468`，starvation `0.3750`。
- 结论：修正版 adaptive 是比初版更平衡的“第 1 + 第 2”组合，UAV sparse 相比 try2 同 seed `B` 的 goodput/PDR 有提升，starvation 低于早期 `B_masked` 的 `0.8750`，queue_slope 仍低于小正式旧 B 参考 `9.7940`。但它不优于 `B_masked_debt_budget` 的吞吐，也还不是论文主表替代方案；更适合作为 density-aware gating 消融或后续调参起点。

- 结论：B' 的证据适合写成“服务债务 + 密度/预算自适应在 sparse 场景显著改善 starvation，并保持接近强传统基线的吞吐/PDR；在 pedestrian 中吞吐接近 B/PPO 且明显高于传统基线”。不能写成 B' 全面优于 B、budget_only、multihead 或 heuristic_only，也不宜替代当前主方法。

### `service_debt_wt_threshold` 软闸门复核

- pilot grid：`logs/benchmark_grid_wt_v5_wt{0,3000,5000,7000}_*`，`edge_toggle` 与 `node_rejoin` 各 1 seed。随着阈值升高，pre 阶段 sent/candidate 比例按 `0.586→0.517→0.481→0.450` 单调下降；`wt<=5000` 时 perturb 阶段仍约 `0.578~0.583`，`wt=7000` 才明显过抑。
- 三 seed 正式复核：`logs/benchmark_dynamic_perturb_v5_wt3000_formal_20260516_003804`。相对原 `wt=0` 动态正式批次，`edge_toggle` 中 `wt=3000` 把 pre sent/candidate 从 `0.583` 压到 `0.522`，扰动期 PDR 跌幅从 `-15.5%` 改善到 `-11.0%`；但 `node_rejoin` 的扰动/恢复 PDR 从 `0.0134/0.1348` 降到 `0.0102/0.1216`，恢复 throughput 从 `2.6072` 降到 `2.3481`。
- 诊断：两个场景在扰动前的 `active_edges` 波动强度几乎相同，`mean(|Δedges|/edges)` 分别约 `0.00575` 与 `0.00558`，`p95` 分别约 `0.0500` 与 `0.0476`。因此“只按 active_edges 变化率自适应 wt”不足以解释 pre 阶段差异，也不是一个已经被数据支持的直接方案。
- 结论：固定 `wt=3000` 只能算场景间折中，不能作为跨动态扰动的新默认。它证明“按 Wt 压制过申请”这条机制有效，但正式结果同时表明阈值需要感知更丰富的运行情境；后续若继续，应优先研究结合请求成功率、队列增长和节点/边联合变化的门控，而不是继续寻找单一全局阈值，或直接照搬仅基于 `active_edges` 变化率的自适应规则。

### `B_masked_debt_context_gate` 情境门控试验

- 新方法入口：`B_masked_debt_context_gate` 继承 `B_masked_debt_adaptive`，固定基线仍用 `service_debt_wt_threshold=3000`，额外记录 `wt_scale / wt_topo / wt_succ / wt_queue`。
- 初版实现（`logs/benchmark_context_gate_smoke_20260516_091850`, `logs/benchmark_context_gate_pilot_v5_20260516_092058`）把 success/queue 压力直接并入 relax。结果在冷启动就出现 `wt_succ≈1.00`、`wt_queue≈1.00`、`wt_scale=0.00`，实质把门控退化为 `wt=0`；pilot goodput/slot 仅 `0.0500/0.0515`，starvation `8.0/6.0`。该批次只说明实现错误，标记 `invalid`。
- 修正版（`logs/benchmark_context_gate_smoke_topology_only_20260516_092252`, `logs/benchmark_context_gate_pilot_v5_topology_only_20260516_092838`）改为仅由显式 topology pressure 放松阈值；success/queue 只保留诊断，不再直接驱动阈值。smoke 中 pre 段 `wt_scale=1.00`，`node_rejoin` 在 `frame>20000` 后按预期转为 `wt_scale=0.00`。
- 单 seed pilot 结果：相对固定 `wt=3000`，`edge_toggle` 总 PDR `0.1142→0.1286`、starvation 维持 `0.5`；但 `node_rejoin` perturb PDR 仅 `0.0128`，低于旧 Wt grid 的全部阈值（`wt=0/3000/5000/7000` 分别为 `0.0440/0.0371/0.0588/0.0483`），throughput 同样只剩 `0.1812`。
- 结论：当前数据不支持“检测到节点缺失后就把 Wt 阈值完全放空”的规则，也不支持把 `B_masked_debt_context_gate` 推进到三 seed 正式验证。若后续继续做 B 路线，应优先研究**有界放松**或更直接的 collision/request 成功反馈，而不是 topology shock 后直接把阈值降到 0。

### `B_constrained_debt` 显式约束首版

- 代码入口：`B_constrained_debt`，在现有 `DynamicTDMA + ablation` 链路上新增显式约束版 RL 入口；不改 C++ MAC 状态机，保留 `twohop` mask、受控 request budget，并把 `service_gap / debt_level / dual_pressure` 作为仅对新方法启用的额外状态输入。
- 训练侧实现：`rl/ppo_trainer.py` 新增 `ConstraintState`、service/starvation cost trajectory、window-level dual update、PPO-Lagrangian 组合 advantage，以及显式 debt 更新；日志新增 `svc_cost / starv_cost / lambda_s / lambda_starv`。
- smoke：`logs/benchmark_constrained_debt_smoke_20260516_20260516_181609`，UAV sparse，seed 1，`target_updates=5`。PPO 正常到达 target，确认新状态维度、cost 轨迹、dual 更新和 benchmark 汇总可用。
- pilot：`logs/benchmark_constrained_debt_pilot_20260516_20260516_181657`，两个 MANET 主场景 × `B B_masked_debt_budget B_constrained_debt` × seed 1，`target_updates=60`。完整性：6/6 `run_status.tsv`，所有 PPO 达到目标 update。
- 关键结果：pedestrian 中 `B_constrained_debt` goodput/PDR `0.2610/0.1276`，略高于 `B` 的 `0.2486/0.1214`，但 queue_slope `18.1791`、Jain 尾100 `0.2127`、starvation `0.6667` 均劣于 `B`。UAV sparse 中则明显失败：goodput/PDR `0.0783/0.0384`，低于 `B` 的 `0.2215/0.1087` 和 `B_masked_debt_budget` 的 `0.2644/0.1295`，starvation 升到 `1.6667`。
- 诊断：按草案初值直接使用 `frames_since_tx > 5` 的 service/starvation cost，会在两个场景都迅速饱和到 `1.000`，日志末帧 `lambda_s/lambda_starv` 仅继续单调上升到 `0.240/0.284`，actor 看不到有区分度的约束信号。当前负结果说明“显式约束主干”已接通，但**约束定义与阈值尚未与系统真实服务节奏对齐**。
- 结论：该版本不能进入正式实验，也不能替代 `B_masked_debt_budget`。下一步优先级应改为：先从现有正式结果反推合理的 service-gap 分布或改用连续 debt-based cost，再重新估计 cost limit；在 cost 不再饱和前，不应继续调高 `lambda_lr` 或直接做多 seed。

### `B_constrained_debt` 连续 cost 重估

- 新增 `scripts/analyze_service_gap.py`，可从既有 `slot_stats.csv + frame_metrics.csv` 重构 queued-node 的真实 `service_gap` 分布，并同步投影连续 cost。正式结果反推显示：`v3 B` 的 pedestrian `p50/p95/p99=51/817/1653`，UAV sparse `51/347/674`；`gap>5` 占比分别约 `67.4% / 76.5%`，因此首版 `threshold=5` 不是“严格”，而是口径失真。
- 约束定义已改为连续 debt cost：`service_cost = clip((debt-threshold)/ramp, 0, 1)`，starvation 同样采用连续 ramp；显式 debt 改成服务间隔风格的有界状态，成功服务后按默认配置清零。最终默认值改为 `service_threshold=80`、`service_ramp=1200`、`starvation_threshold=300`、`starvation_ramp=1800`、`debt_max=2000`。
- 用正式 `B` 批次投影新口径后，pedestrian 的 `service/starvation cost` 均值约 `0.1169/0.0442`，UAV sparse 约 `0.0435/0.0075`；据此把默认 limit 重估为 `service_cost_limit=0.10`、`starvation_cost_limit=0.03`。这组值低于 pedestrian 当前水平，但没有把 sparse 逼向接近零的不可实现目标。
- 复核过程分两轮：中间版 `logs/benchmark_constrained_debt_continuous_smoke_20260516_20260516_184141` + `logs/benchmark_constrained_debt_continuous_pilot_20260516_20260516_184211` 把二值 cost 改成连续 cost，但 `service_ramp=320 / debt_max=400` 仍低于正式 pedestrian 的 `p95`，训练后半段继续饱和；因此只保留为诊断批次。最终版 `logs/benchmark_constrained_debt_retuned_smoke_20260516_20260516_185013` 的前 5 个 update 中 `svc_cost` 依次约 `0.007/0.092/0.199/0.306/0.412`，说明“开局即饱和”问题已解除。
- 最终 pilot：`logs/benchmark_constrained_debt_retuned_pilot_20260516_20260516_185034`，两个 MANET 主场景 × `B B_masked_debt_budget B_constrained_debt` × seed 1，6/6 run 完成。pedestrian 中新方法 goodput/PDR `0.2855/0.1398` 高于 `B` 的 `0.2638/0.1295`，queue_slope 也从 `16.3824` 降到 `15.0241`，但 starvation `0.6667` 仍劣于 `B` 与 `B_masked_debt_budget` 的 `0.5833`。UAV sparse 中新方法 goodput/PDR `0.2501/0.1223` 高于 `B` 的 `0.2265/0.1111`，但低于 `B_masked_debt_budget` 的 `0.2780/0.1362`，且 starvation `0.7500` 劣于另外两者。
- 结论：本轮已完成“反推分布 → 连续 cost → 重估 limit → 同配置复跑 pilot”的实现闭环。新口径修复了首版的定义错误，但**没有把 `B_constrained_debt` 变成更优主方法**；训练后半段仍会进入高债务饱和区，说明问题已从“cost 口径错误”转为“当前执行层/预算结构下仍会形成长期饥饿”。现阶段它仍只应保留为 exploratory 约束分支，不进入正式主表。

### `B_constrained_debt` 结构拆分诊断

- 为回答“two-hop mask + bounded budget 是否把部分节点锁进长期失服”“tail dual 是否优于全局均值”“是否应先只让 constrained 分支接管 budget”三件事，新增 `scripts/analyze_tail_service.py`，并补充四个对照入口：`B_constrained_debt_nomask`、`B_constrained_debt_nobudget`、`B_constrained_tail_dual`、`B_constrained_budget_only`。
- smoke：`logs/benchmark_constrained_debt_variant_smoke_20260516_20260516_190949`，五个分支均跑通。pilot：`logs/benchmark_constrained_debt_variant_pilot_20260516_20260516_191055`，两个 MANET 主场景 × 五方法 × seed 1，10/10 run 完成，全部达到 `target_updates=60`。
- 节点级诊断不支持“two-hop mask 单独把节点锁死”的解释。pedestrian 中当前 `B_constrained_debt` 的 long-gap 总量/最差 gap 为 `524 / 1365`；去 mask 后为 `778 / 1523`，并未解除尾部；去 budget 后恶化到 `4547 / 3156`；tail dual 也恶化到 `3893 / 2585`。UAV sparse 中去 mask 后 long-gap 归零，但去 budget 仍把单节点拖到 `546 / 1745`，说明 budget 更像保护极端尾部的边界，而 mask 主要改变冲突结构。
- tail dual 没有兑现“更贴近尾节点”的预期。UAV sparse 中 `B_constrained_tail_dual` 直接退化到 goodput/PDR `0.0526/0.0258`、starvation `3.5000`；节点级仍出现 `406` 个 long-gap 帧。把 dual 统计量换成 top-tail，并不会自动产生可执行的逐节点补偿动作。
- `budget_only` 是本轮最值得保留的简化分支，但还不是新主方法。pedestrian 中它取得本轮最高 goodput/PDR `0.2740/0.1339` 与较低 starvation `0.4167`，但 queue_slope 升到 `19.1619`、节点级 long-gap 总量仍有 `3358`；UAV sparse 中它给出本轮最平衡的 `0.2736/0.1338`、queue_slope `7.6433`、starvation `0.5000`，且 long-gap 归零。
- 结论：本轮已经排除了两个看似直接、但数据不支持的方向：**不应默认移除 request budget，也不应继续推进当前 tail-dual 聚合**。下一步若继续 constrained 路线，应围绕 `budget_only` 做更直接的 per-node action allocation / debt repayment 设计，而不是继续在全局 dual 聚合器上做小修小补。

### `B_constrained_tail_budget` 硬尾部预算试验

- 新方法入口：`B_constrained_tail_budget`。它继承 `budget_only` 的简化结构，但把预算加成从“每节点按自身 debt 独立放宽”改成“同帧只给 top-tail debt 节点预算 bonus”，用 `constraint_budget_allocation=tail_ranked` + `constraint_tail_fraction=0.25` 做硬分配。
- smoke：`logs/benchmark_constrained_tail_budget_smoke_20260517_20260517_102230`，UAV sparse，`B_constrained_budget_only` 与新分支都达到 `target_updates=5`。新分支可跑通，但 smoke 已显示 queue_slope `12.0079` 高于 `budget_only` 的 `9.4931`。
- pilot：`logs/benchmark_constrained_tail_budget_pilot_20260517_20260517_102311`，两个 MANET 主场景 × `B_constrained_debt / B_constrained_budget_only / B_constrained_tail_budget` × seed 1，6/6 run 完成且都达到 `target_updates=60`。
- 结果：pedestrian 中 `tail_budget` goodput/PDR 仅 `0.2225/0.1090`，低于 `budget_only` 的 `0.2653/0.1300`，starvation 也从 `0.4167` 恶化到 `1.0000`；UAV sparse 中更直接失效，goodput/PDR `0.0477/0.0234`、starvation `4.0000`，远差于 `budget_only` 的 `0.2668/0.1306`、`0.5000`。
- 节点级诊断说明它不是“没起作用”，而是“起作用的方式不对”。pedestrian 中若干节点的 `sent_per_candidate` 被压到 `0.41~0.50`，另一些节点仍在 `0.76` 左右，long-gap 总量仍有 `4881`；UAV sparse 中也出现明显请求份额分裂，且重新出现 long-gap 节点。硬 top-k 分配会把公平问题从“大家一起高压”变成“被选中者与未选中者之间的新不公平”。
- 结论：**不继续推进硬 tail-ranked budget**。若 constrained 路线还要继续，下一步应改成连续、保底的预算共享机制，例如所有节点保留 minimum share，再按 debt 平滑增配；或者直接做逐节点/逐时隙的债务还款动作，而不是用硬 top-k 裁掉一批节点的 bonus。

### `B_constrained_smooth_budget` 连续预算共享试验

- 新方法入口：`B_constrained_smooth_budget`。它仍基于 `budget_only`，但把硬 top-k 改成 `constraint_budget_allocation=smooth_ranked`：所有有债节点保留 `constraint_budget_min_share=0.50`，再按 debt 排名平滑增配，并保持同帧平均 budget level 基本不变。
- smoke：`logs/benchmark_constrained_smooth_budget_smoke_20260517_20260517_104853`，UAV sparse，`smooth_budget` goodput/PDR `0.2576/0.1263`、starvation `0.3333`，优于同批 `budget_only` 的 `0.2274/0.1115`、`0.6667`，说明连续分配至少消除了硬切分配的早期崩溃。
- pilot：`logs/benchmark_constrained_smooth_budget_pilot_20260517_20260517_104951`，两个 MANET 主场景 × `B_constrained_debt / B_constrained_budget_only / B_constrained_smooth_budget` × seed 1，6/6 run 完成且都达到 `target_updates=60`。
- 结果：pedestrian 中 `smooth_budget` goodput/PDR `0.2922/0.1426`，接近 `budget_only` 的 `0.2981/0.1457`，queue_slope `16.6812` 略低、Jain 尾100 `0.4700` 更高，但 starvation `0.6667` 反而高于 `0.5000`；UAV sparse 中 starvation 仅从 `0.4167` 小降到 `0.4000`，goodput/PDR 却退到 `0.1395/0.0680`，低于 `budget_only` 的 `0.2410/0.1179`。
- 节点级诊断说明“连续”并没有自动带来更好的尾部。pedestrian 中 `smooth_budget` 的 long-gap 总量/最差 gap 为 `2621 / 2086`，仍高于 `budget_only` 的 `1584 / 1718`；UAV sparse 中 `budget_only` 没有 long-gap，而 `smooth_budget` 新增 `458` 个 long-gap 帧、最差 gap `1657`。它比硬 top-k 温和，但仍把预算重排成了对部分节点不利的软失衡。
- 结论：**当前连续分配版也不升为默认**。若继续 constrained 路线，下一步不应再只围绕 rank-based budget weighting 微调；更值得试的是直接把“还债”写进动作/调度执行层，或引入能同时约束吞吐损失与尾节点补偿的局部反馈。

### `B_constrained_repay_action` 显式还债动作试验

- 新方法入口：`B_constrained_repay_action`。它沿用 `budget_only` 的 request-budget 结构，但新增 `constraint_repay_action_slots=1`：每帧最多为一个高债 tail 节点保留一个安全槽位，并把该 deterministic override 从 PPO actor mask 中剔除，避免把执行层强制动作误当成采样动作训练。
- smoke：过激首版 `logs/benchmark_constrained_repay_action_smoke_20260517_20260517_163955` 曾在 UAV sparse 退化到 goodput/PDR `0.0592/0.0289`、starvation `3.0000`；修正 `constraint_repay_action_max_nodes` 生效位置后，`logs/benchmark_constrained_repay_action_smoke_top1_fix_20260517_20260517_164159` 的 `repay_slots` 已收敛为每窗 `84/128/128`，说明新版确实只接管 top-1 债务节点。
- pilot：`logs/benchmark_constrained_repay_action_pilot_20260517_20260517_164231`，两个 MANET 主场景 × `B_constrained_debt / B_constrained_budget_only / B_constrained_repay_action` × seed 1，6/6 run 完成且都达到 `target_updates=60`；两个场景合计 120 个 PPO update 中共触发 `4447` 次 forced repayment，平均 `37.1` 次/update。
- 结果：pedestrian 中 `repay_action` 把 starvation 从 `budget_only` 的 `0.7778` 降到 `0.5000`，queue_slope 也从 `15.7423` 降到 `11.1240`，但 goodput/PDR 退到 `0.2307/0.1127`；UAV sparse 中 goodput/PDR `0.2590/0.1264` 接近 `debt`，却没有优于 `budget_only` 的 starvation，仍为 `0.5833`。
- 节点级诊断否定了“显式动作必然修复真实 tail”的预期。考虑异步停机带来的 run 长度差异后，pedestrian 的 long-gap rate 仍是 `repay_action 5.84% > budget_only 3.76% > debt 0.75%`，且最差 gap 升到 `2961`；UAV sparse 虽把 long-gap rate 从 `debt` 的 `0.066%` 降到 `0.038%`，但仍不如 `budget_only` 的 `0%`。
- 结论：**当前显式还债动作也不升为默认**。它能把一部分表层 starvation 指标压下来，但会用较强的执行层 override 改写动作分布，且没有稳定换来更好的真实 tail；若继续探索，应先定义更局部、可学习、受吞吐约束的 repayment 目标，而不是继续扩大 deterministic override。

### `B_constrained_repay_reward` 局部成功奖励试验

- 新方法入口：`B_constrained_repay_reward`。它仍基于 `budget_only`，但不再强制改写动作，而是尝试把动作时刻的局部 tail-debt credit 注入成功发送 reward，希望让 actor 自己学习“为欠债节点还款”。
- smoke：先后跑了 `logs/benchmark_constrained_repay_reward_smoke_20260517_20260517_170649` 与 `logs/benchmark_constrained_repay_reward_cost_smoke_20260517_20260517_170828`。第二版已把 credit 从原始 debt 改成动作时刻 cost，并进一步约束到 same-frame top-tail 节点。
- 结果：两版 smoke 的 5 个 update 中 `repay_bonus` 都始终为 `0.000`。这不是“奖励太小”，而是成功事件与局部 tail credit 的交集几乎没有出现，说明 success-only reward 无法在当前冷启动分布下为尾节点提供有效 bootstrap。
- 结论：**不把 `repay_reward` 推进到 pilot**。若局部目标只能在成功以后才给信号，而尾节点恰恰长期得不到成功，它就无法解决当前问题。

### `B_constrained_repay_target` 局部软目标试验

- 新方法入口：`B_constrained_repay_target`。它继续保留 `budget_only` 的执行逻辑，只对 same-frame top-tail 节点增加一个有界辅助损失：当安全槽最大申请概率低于 `constraint_repay_target_prob=0.75` 时，按 `constraint_repay_target_coef=0.05` 惩罚短缺，不再等待先出现成功发送。
- smoke：`logs/benchmark_constrained_repay_target_smoke_20260517_20260517_171055` 与收窄到 top-tail credit 后的 `logs/benchmark_constrained_repay_target_tail_smoke_20260517_20260517_171156` 都跑通；后者 `repay_target` 从 `0.0003` 增到 `0.0029`，说明辅助目标确实被接入，但强度较小。
- pilot：`logs/benchmark_constrained_repay_target_pilot_20260517_20260517_171235`，两个 MANET 主场景 × `B_constrained_debt / B_constrained_budget_only / B_constrained_repay_target` × seed 1，6/6 run 完成并达到 `target_updates=60`。两个场景合计 120 个 update 的平均 `repay_target` 仅 `0.000726`。
- 结果：pedestrian 中 `repay_target` 把 queue_slope 从 `budget_only` 的 `16.9343` 压到 `11.8849`，但 goodput/PDR 退到 `0.2443/0.1200`，starvation 仍是 `0.5833`；UAV sparse 中 goodput/PDR 升到 `0.2709/0.1323`、queue_slope 降到 `7.7121`，但 starvation 升到 `0.5833`，没有优于 `budget_only`。
- 真实 tail 仍然否定该方向。按 queued-frame 归一化后，pedestrian 的 long-gap rate 为 `repay_target 2.3266% > budget_only 2.0340% > debt 1.0878%`，最差节点 9 的 long-gap 帧达到 `2702`、最差 gap `3183`；UAV sparse 也从 `budget_only/debt` 的 `0` 新增到 `31` 个 long-gap 帧。
- 结论：**当前局部软目标也不升为默认**。它说明“可学习信号”比 success-only reward 更容易接上，但当前局部 tail credit 仍与真实服务机会不对齐；继续做 per-node 小修补的边际价值已经很低，下一步更值得转向 tail-risk / CVaR 或 centralized critic 一类能直接看见全局尾部结构的方案。

### `B_constrained_shared_actor` / `B_constrained_mappo` CTDE 验证

- 新入口：`training_mode=independent/shared_actor/mappo`。`shared_actor` 使用共享本地 actor + 共享局部 critic；`mappo` 保持 actor 只读本地观测，但让 critic 读取按 `node_id=0..11` 固定排序的全网状态序列，并以目标节点 one-hot 条件化输出逐时隙 value。共享 checkpoint 已单独保存 `training_mode / actor_state / critic_state`，旧 independent checkpoint 保持兼容。
- smoke：`logs/benchmark_constrained_mappo_smoke_20260517_20260517_190432`，UAV sparse 上 `shared_actor` 与 `mappo` 都达到 `target_updates=5`，共享训练闭环、checkpoint 和 centralized critic loss 均正常。
- pilot：`logs/benchmark_constrained_mappo_pilot_20260517_20260517_190512`，两个 MANET 主场景 × `B_constrained_budget_only / B_constrained_shared_actor / B_constrained_mappo` × seed 1，6/6 run 完成并达到 `target_updates=60`。分析中发现 `target_updates` 提前停机时 `slot_stats.csv` 末帧可能只写到部分节点，已修复 `scripts/summarize_ablation_metrics.py` 为使用“最后一个完整节点帧”重汇总。
- 结果：pedestrian 中 `mappo` 把 long-gap rate 从 `budget_only` 的 `1.1220%` 降到 `0.9759%`，goodput/PDR 仍接近 `0.2974/0.1458` 对 `0.3026/0.1480`，但 queue_slope 从 `17.1714` 升到 `18.2536`，超过本轮 `5%` 回退门槛；`shared_actor` 则退到 long-gap rate `3.1745%`。UAV sparse 中 `mappo` 把 long-gap 从 `1047` 帧压到 `0`，starvation 从 `0.5000` 降到 `0.2500`，goodput/PDR 提升到 `0.2612/0.1281`，但 queue_slope 也从 `10.9904` 升到 `12.4040`；`shared_actor` 虽把 long-gap 降到 `122`、starvation 降到 `0.2500`，但吞吐/PDR 仅基本持平。
- 结论：**CTDE 第一版证明 centralized critic 能更直接地看见 tail 结构，但当前还没有通过 pilot 门槛**。它比前几轮局部 debt 小修补更接近目标，尤其在 UAV sparse 中明显修复 long-gap；但 queue stability 仍未同步改善，因此本轮不进入 `3 seeds × 400 updates` 小正式。下一步若继续，优先研究 critic / advantage 是否过度偏向服务均衡，或直接把 tail-risk 与 queue-risk 做成联合 critic 目标，而不是立刻扩大实验规模。

### `B_constrained_mappo_risk` 风险显式化验证

- 新入口：`training_mode=mappo_risk`。actor 仍只读本地观测；centralized critic 在原全网状态矩阵上，为每个节点显式追加 `tail_risk=max(service_cost, starvation_cost)` 与基于 backlog/正向 queue-growth EWMA 的 `queue_risk`，并追加帧级 mean/top-fraction tail-risk 与 mean/top-fraction queue-risk。
- smoke：`logs/benchmark_constrained_mappo_risk_smoke_20260517_20260517_215909`，UAV sparse 上原版 `mappo` 与 `mappo_risk` 都达到 `target_updates=5`；`mappo_risk` 的早期 goodput/PDR 为 `0.2434/0.1195`，queue_slope `9.2160`，优于同批 `mappo` 的 `0.2209/0.1087` 与 `13.0013`，说明新特征已进入 critic 并会改变训练轨迹。
- pilot：`logs/benchmark_constrained_mappo_risk_pilot_20260517_20260517_215950`，两个 MANET 主场景 × `B_constrained_budget_only / B_constrained_mappo / B_constrained_mappo_risk` × seed 1，6/6 run 完成并达到 `target_updates=60`。
- 结果：pedestrian 中 `mappo_risk` 相比 `budget_only` 把 long-gap rate `3.3073%→2.5124%`、最差 gap `3600→1837`，queue_slope 也略降 `19.0825→18.4799`；但它不如原版 `mappo` 的 long-gap rate `1.9096%`，且 goodput/PDR 降到 `0.2544/0.1243`、starvation 仍为 `0.8333`。UAV sparse 中 `mappo_risk` 消除了原版 `mappo` 新引入的 `15` 个 long-gap 帧，goodput/PDR 提升到 `0.2629/0.1288`，queue_slope 也较 `mappo` 从 `10.5221` 降到 `8.6565`；但相对 `budget_only` 的 `6.3230` 仍回退 `36.9%`，starvation 也从 `0.3333` 升到 `0.4167`。
- 结论：**显式 tail-risk + queue-risk critic 特征有效改变了 CTDE 的权衡，但仍未通过 pilot 门槛**。它在 UAV sparse 上缓和了原版 MAPPO 的 queue/tail 冲突，却没有同时优于 `budget_only`；在 pedestrian 上则出现“尾部比基线好、但均值与 starvation 未跟上”的折中。下一步若继续，应优先改 critic 目标或 advantage weighting，而不是只继续堆输入特征。

### `B_constrained_mappo_joint_value` 联合价值目标验证

- 新入口：`training_mode=mappo_joint_value`。它继续复用 `mappo_risk` 的 critic 输入，但不再用 reward-only GAE；共享 rollout 会直接按 `reward - joint_value_tail_coef * tail_risk - joint_value_queue_coef * queue_risk` 计算 advantage 与 return。第一版固定 `tail/queue=0.5/0.5`，目的是验证“改目标”是否比“只加特征”更接近指标。
- smoke：`logs/benchmark_constrained_mappo_joint_value_smoke_20260517_20260517_222205`，UAV sparse 上 `joint_value` 相比同批 `mappo_risk` 把 queue_slope 从 `11.8710` 降到 `8.4222`，说明联合目标已生效；后续补跑 `logs/benchmark_constrained_mappo_joint_value_tailheavy_smoke_20260517_20260517_223507`，把 tail 权重提到 `1.0/0.5` 后 early queue_slope 反升到 `10.2866`、starvation 升到 `0.6667`，没有显示出更好的开局。
- pilot：`logs/benchmark_constrained_mappo_joint_value_pilot_20260517_20260517_222300`，两个 MANET 主场景 × `B_constrained_budget_only / B_constrained_mappo_risk / B_constrained_mappo_joint_value` × seed 1，6/6 run 完成并达到 `target_updates=60`。
- 结果：pedestrian 中 `joint_value` 把 long-gap rate 从 `budget_only 4.2272%` 压到 `1.9607%`，最差 gap `4900→1896`，且略优于 `mappo_risk 2.0896%`；goodput/PDR `0.2537/0.1242`、queue_slope `18.7466` 相比基线 `0.2586/0.1268`、`18.4349` 均在 `5%` 门内，但 starvation 仍只是持平 `0.6667`。UAV sparse 中 `joint_value` 把 goodput/PDR 提升到 `0.2756/0.1349`，queue_slope 从 `budget_only 11.3145` 降到 `9.1623`，starvation 从 `0.5000` 降到 `0.2500`；但它从基线与 `mappo_risk` 的 `0` 新增了 `286` 个 long-gap 帧，long-gap rate `0.2008%`，最差 gap `1441`。
- 结论：**联合价值目标比“只给 critic 加特征”更接近真正的多目标优化，但当前单标量 `joint return` 仍未通过门槛**。它第一次在 UAV sparse 上同时改善了吞吐、PDR、queue 与 starvation，却把 long-gap 重新放出来；这说明下一步不应继续只调一组线性权重，更值得把 critic 拆成 reward/tail/queue 多头，或用 tail-aware advantage gating 让尾部风险拥有不可被 queue 收益抵消的通道。

### `B_constrained_mappo_multihead` / `B_constrained_mappo_tail_gate` 多通道价值验证

- 新入口：`training_mode=mappo_multihead` 使用共享 backbone + `reward/tail/queue` 三头 centralized critic，并分别学习三套 return；`training_mode=mappo_tail_gate` 保留单头 critic，但让 actor advantage 在 tail 压力为正时额外受 `tail_gate_coef` 下界约束，避免 queue 收益把 tail 风险完全抵消。两者都继续保持 actor 只读本地观测。
- smoke：`logs/benchmark_constrained_mappo_heads_gate_smoke_20260517_20260517_225904`，UAV sparse 上 `joint_value / multihead / tail_gate` 3/3 达到 `target_updates=5`；其中 `tail_gate` 的 early queue_slope `9.1172`、starvation `0.4167` 最好，说明门控会明显改变优化方向。
- pilot：`logs/benchmark_constrained_mappo_heads_gate_pilot_20260517_20260517_230005`，两个 MANET 主场景 × `B_constrained_budget_only / B_constrained_mappo_joint_value / B_constrained_mappo_multihead / B_constrained_mappo_tail_gate` × seed 1，8/8 run 完成并达到 `target_updates=60`。
- 结果：pedestrian 中 `multihead` 的 long-gap rate `1.6707%` 略低于同批 `budget_only 1.6932%`，但 goodput/PDR 退到 `0.2426/0.1190`、queue_slope 升到 `20.4056`；`tail_gate` 更差，long-gap rate 升到 `2.6176%`。UAV sparse 中 `multihead` 把 `joint_value` 的 long-gap 帧从 `592` 压到 `58`，同时 goodput/PDR `0.2676/0.1312`、queue_slope `9.1164` 都优于同批基线；`tail_gate` 则把 long-gap 保持为 `0`，queue_slope 压到 `4.8366`，starvation 从 `0.5000` 降到 `0.3333`，但这种收益没有迁移到 pedestrian。
- 结论：**多头 critic 比单标量 joint return 更能分离 tail 与 queue credit，tail-aware gating 则呈现明显场景依赖**。当前 `multihead` 是更值得保留的结构方向，但还没有通过 pilot：pedestrian 均值指标回退超过 `5%`，UAV sparse 也未完全追平基线的 `0` long-gap。`tail_gate` 在 sparse 场景很强，却在 pedestrian 过度收紧，暂时更像诊断工具而不是可升级候选。

### `mappo_multihead` actor 解耦修正（2026-05-17）

- 病灶定位：`B_constrained_mappo_multihead` pilot 在 `scripts/ablation.sh` 里被默认设成 `joint_value_tail_coef=0.5, joint_value_queue_coef=0.5`，而 `rl/ppo_trainer.py` 的 multihead 分支直接复用了 `mappo_joint_value` 的两个系数，导致 actor advantage 在每次 PPO 更新时都按 `reward - 0.5·tail - 0.5·queue` 等权合成。pedestrian 场景里 tail 信号本就稀薄，等权混入相当于稀释了 reward 通道的梯度方向，正是 goodput/PDR 退到 `0.2426/0.1190`、queue_slope 涨到 `20.41` 的直接原因。
- 修正：在 `PPOConfig` 中新增 `multihead_actor_tail_coef` / `multihead_actor_queue_coef`（默认 `0.0`），把"critic 学三头价值"和"actor 是否吸纳 tail/queue advantage"在配置层彻底拆开。`ppo_trainer.py` 的 multihead 分支不再读取 `joint_value_*`，改为读取新系数；当两者都为 0 时直接 `combined_advantages = advantages`，等价于"critic 多头 + actor 单标量 reward"。`scripts/ablation.sh` 的 `B_constrained_mappo_multihead` 默认改为 `MULTIHEAD_ACTOR_TAIL_COEF=0 / MULTIHEAD_ACTOR_QUEUE_COEF=0`，新 pilot 起点即为 actor-decoupled，旧 0.5/0.5 路径需要显式传 env var 才能复现。

- 校验：`python3 -m py_compile rl/ppo_trainer.py rl/rl_agent.py`、`bash -n` 三个脚本均通过；`argparse` 接入测试 `0.25/0.10` OK；ablation dry-run 在默认/env-override 两种路径下都正确输出 `multihead_actor=0/0` 或 `0.2/0.1`。

|:---|:---|:---|

| `B_constrained_budget_only` | constrained-RL 简化候选 | 本轮最稳的探索分支；可保留作下一轮起点，但仍未达到主方法标准 |
| `B_constrained_tail_budget` | constrained-RL 负结果诊断 | 硬 top-tail 预算重分配会制造新的未服务节点，已被 pilot 否定 |
| `B_constrained_smooth_budget` | constrained-RL 负结果诊断 | 连续 rank-based 共享比硬 top-k 温和，但 pilot 仍劣于 `budget_only`，暂不继续推进 |
| `B_constrained_repay_action` | constrained-RL 负结果诊断 | 显式 top-1 还债动作能压低部分表层 starvation，但真实 tail 未稳定改善，暂不继续推进 |
| `B_constrained_repay_reward` | constrained-RL 负结果诊断 | success-only 局部还债奖励在 smoke 中未能激活，无法为 tail 节点 bootstrap |
| `B_constrained_repay_target` | constrained-RL 负结果诊断 | top-tail 局部软目标虽能接入训练，但 pilot 真实 tail 仍劣于 `budget_only`，暂不继续推进 |
| `B_constrained_shared_actor` | CTDE 中间对照 | 参数共享本身可改善 UAV sparse tail，但 pedestrian 回退，不能单独解释最终收益 |
| `B_constrained_mappo` | CTDE exploratory | centralized critic 已能压低 tail，尤其修复 UAV sparse long-gap；当前因 queue_slope 未过门槛，暂不进入正式批次 |
| `B_constrained_mappo_risk` | CTDE exploratory | 显式 tail/queue risk critic 能缓和原版 MAPPO 的部分冲突，但仍未同时满足 tail、queue 与吞吐门槛 |
| `B_constrained_mappo_joint_value` | CTDE exploratory | 联合 tail/queue return 能显著改善 queue 与 starvation，但 UAV sparse 重新出现 long-gap，说明单标量联合目标仍不足 |
| `B_constrained_mappo_multihead` | CTDE exploratory | actor 解耦后单 seed sweep 表现最好，但 2026-05-23 三 seed 主表复核未过门：pedestrian 低于 `B`，UAV sparse 低于 `budget_only` 且 queue/starvation 不达标 |
| `B_constrained_mappo_tail_gate` | CTDE exploratory | tail-aware advantage gating 能在 UAV sparse 同时守住 tail 与 queue，但 pedestrian 出现过度收紧，当前只保留为诊断分支 |

## 真实论文支撑方向

- STDMA/spatial-reuse TDMA：用于支撑 `greedy_stdma` 与 `greedy_stdma_2hop` 的对照动机，写作时强调冲突图、空间复用和集中/启发式调度思想。
- Z-MAC：用于支撑 `zmac_inspired` 的混合 CSMA/TDMA 与 owner-slot 思想。
- TRAMA：用于支撑 `trama_inspired` 的流量感知、邻居信息和时隙分配思想。
- DLMA/DQN-MAC：更适合作为 related work；若论文需要更强 RL 同类实验，再实现轻量 DQN access baseline。
- B-MAC/S-MAC/T-MAC：由于当前项目没有真实能耗模型，适合 related work，不优先进入主实验表。

## 论文写法约束

- 不把 v4 写成全面优于 v3。
- 不把 `zmac_like`、`trama_like`、`zmac_inspired`、`trama_inspired` 写成原论文协议的 bit-level 复现。
- 不做能耗优越性结论，除非后续补齐真实能耗模型。
- 论文主张应聚焦：在 MANET/UAV-inspired 动态拓扑与队列压力下，RL 调度比静态/启发式 TDMA 更能适应负载；饥饿惩罚可改善 fairness，但存在吞吐权衡。
