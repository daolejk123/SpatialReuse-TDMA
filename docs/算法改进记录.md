## 问题 1：奖励崩溃（策略退化）

**发现时间**：2026-03-15 闭环训练首次验证

**现象**：
训练初期 `avg_r` 约 +44（与启发式策略持平），但随训练推进迅速下降至负值：

```
frame=   32  avg_r=+44.367  entropy=0.6926
frame=  352  avg_r=+47.134  entropy=0.6886   ← 短暂上升
frame=  960  avg_r=+28.230  entropy=0.5076   ← 开始下降
frame= 1184  avg_r=+0.457   entropy=0.4229
frame= 1664  avg_r=-91.629  entropy=0.3823   ← 严重崩溃
frame= 1920  avg_r=-197.371 entropy=0.3038
```

**分析**：
1. **熵坍缩**：`entropy` 从 0.693（均匀随机）快速降至 0.30，策略过早收敛到确定性动作
2. **协作失败**：多节点独立学习同一个策略网络（共享参数），容易收敛到"所有节点选同一时隙"的局部最优，导致大量冲突
3. `ent_coef=0.01` 过小，不足以维持探索

**可能的改进方向**：

### 方向 A：增大熵正则化
```python
# 当前
ent_coef = 0.01

# 建议尝试
ent_coef = 0.05  # 或 0.1
```
熵正则化系数决定了策略"愿意保持多大随机性"。增大后策略会更慢收敛，但可避免过早陷入差的局部最优。

### 方向 B：降低学习率
```python
# 当前
lr = 3e-4

# 建议尝试
lr = 1e-4  # 更保守的更新步长
```

### 方向 C：增加 GAE（广义优势估计）
当前 advantage 计算是简单的单步 TD：`A_t = r_t + γ·V(t+1) - V(t)`。引入 GAE(λ) 可以减小方差：
```python
# RolloutBuffer.compute_advantages() 中替换为 GAE
gae = 0.0
for t in reversed(range(len(traj))):
    delta = traj[t].reward + gamma * next_val - traj[t].value
    gae = delta + gamma * gae_lambda * gae
    traj[t].advantage = gae
    traj[t].ret = gae + traj[t].value
```

### 方向 D：节点 ID 嵌入（区分各节点策略）
当前所有节点共享同一组参数且输入中没有节点 ID 信息，导致策略无法区分不同节点。可在状态向量中加入 one-hot 节点 ID 编码或可学习嵌入，使不同节点能学习差异化策略。

## 问题 4：ActionSender 连接延迟（frame 3072 才生效）

**发现时间**：2026-03-29 第一次完整闭环验证

**现象**：
```
[ActionSender] 动作管道暂未连接（C++ 仿真未启动），将延迟重试
...（3072 帧后）...
[ActionSender] 动作管道已连接
```
前 3072 帧 C++ 使用启发式策略，RL 网络在"启发式分布"上学习。
闭环一生效，分布发生偏移，avg_r 从 +89 开始缓慢下降，最终在 frame 4800 崩溃。

**分析**：
- Python 先于仿真启动，`ActionSender.open()` 以 O_NONBLOCK 尝试打开动作管道写端
- 此时 C++ 尚未打开读端 → ENXIO 失败，`_fd = None`
- C++ 在 `initialize()` 中打开 action 管道读端，但 Python 仅在 `send()` 时重试连接
- `send()` 每帧调用一次，重试逻辑在 send() 内部，最终在某帧连接成功

**核心问题**：前期训练数据由启发式策略生成，策略网络学习了错误的状态-价值映射，闭环后发生 off-policy 偏差。

**修复方案**（2026-03-29）：
- 在 `rl_receiver.py` 的 `connect()` 中尽早创建 ActionSender 实例并尝试连接
- C++ 端在 `initialize()` 阶段就打开 action 管道读端，确保 Python 首次 `send()` 时能立即连接
- 修复后 RL 动作从 frame 1 开始生效，消除前期 off-policy 偏差

## 问题 5：L_critic 全程偏高（300-1000+）

**发现时间**：2026-03-29 同次验证

**现象**：
```
frame=   64  L_critic=372.6581
frame= 1920  L_critic=358.2158
frame= 4864  L_critic=382.4927  ← 崩溃开始后急剧升高
frame= 5696  L_critic=1058.4805
```

**分析**：
- avg_r 约 +89，returns 约为同量级（假设 γ=0.99，长期 return ≈ r/(1-γ) = 8900）
- Critic 的 MSE Loss = (V - R)² 量级自然很大
- 缺少奖励归一化（reward normalization），导致 Critic 难以收敛

**修复方案**（2026-03-29）：
在 `ppo_update()` 中对 returns 进行归一化，与 advantages 处理方式一致：
```python
# 在 ppo_update() 中，advantages 已归一化，对 returns 也做归一化
returns = (returns - returns.mean()) / (returns.std() + 1e-8)
```

修复后 L_critic 从 300-1000+ 降至 1-10 量级，Critic 收敛速度显著提升。

## 问题 2：前 50 帧无 RL 动作（冷启动延迟）

**现象**：
C++ 从 frame 53 才开始收到 RL 动作，前 52 帧使用启发式策略。

**分析**：
- Python 先启动 → `ActionSender.open()` 因 C++ 尚未启动而失败
- C++ 启动后打开 action 管道读端
- Python 在第一帧 `send()` 中重连成功
- 但 RL 动作基于上一帧观测，所以实际上从 frame 2 就应该有动作
- 实际延迟约 50 帧，可能由 OMNeT++ 仿真速度快于 Python 处理速度导致

**影响**：轻微。前 50 帧使用启发式策略，对长期训练影响不大。

**可能的改进**：C++ 端在写完 state 管道后可短暂等待（如 1ms），给 Python 处理时间。但这会降低仿真速度，需权衡。

## 问题 3：（待观察）on-policy 纯度

**潜在问题**：
PPO 是 on-policy 算法，要求训练数据来自当前策略。但由于管道异步性和 C++ 仿真速度，可能出现：
- C++ 使用的是 frame N 的 RL 动作
- Python 训练使用的是 frame N+K 的观测（K 帧延迟）

这会引入 off-policy 偏差，可能影响训练稳定性。

**可能的改进**：
- 引入同步机制（C++ 等待 Python 返回动作后再继续）
- 或者切换到 off-policy 算法（如 SAC），天然容忍数据延迟

## 问题 6：通信性能指标系统评估（2026-04-11）

**发现时间**：2026-04-11 通信性能日志检测

### 测量方法

仿真结束后分析 `results/` 目录下三类 CSV（本次时间戳 `20260411_222713`，71 帧）：
- `slot_stats_*.csv`：累计请求数 / 成功传输数 / 实际包数
- `frame_metrics_*.csv`：逐帧每节点 Wt（队头等待）、Qt（队长）、Cctrl（控制冲突）、Jlocal（局部公平性）
- `fairness_*.csv`：全局 Jain 指数逐帧趋势

### 性能数据（Apr 11 vs Mar 29 对比）

| 指标 | Apr 11（RL 崩溃后） | Mar 29（RL 高峰期） | 目标 |
|:--|:--|:--|:--|
| 全网 Tx 成功率 | **59.3%** | 50.9% | >70% |
| 平均 Jain 公平性 | 0.591 | 0.597 | >0.85 |
| 达标帧（Jain≥0.85） | 2.8%（2/71） | 1.4%（1/71） | >50% |
| 平均队头等待 Wt | 0.68s | 0.65s | <0.05s |
| 全网平均控制冲突 Cctrl | **0.57** | 0.80 | ≈0 |

### 问题分析

**1. Jain 公平性严重不足（核心问题）**

Jain 指数均值约 0.59，每帧在 0.19~0.90 之间剧烈波动，大多数帧远低于 0.85 目标。
原因分析：
- **拓扑不对称**：Node 0 仅连 1 个邻居，Node 3 连 4 个，竞争强度天然不等。Node 1/3/6/8 因邻居多导致冲突率（Cctrl 0.73~1.72）显著高于 Node 0/2/4/5/7（Cctrl ≈ 0）。
- **RL 公平性激励不足**：原 `r_gamma=0.3` 公平性权重偏低，策略倾向于最大化吞吐量而忽视公平性。

**2. 队头等待时间严重超标**

所有节点平均 Wt 在 0.6~1.3s（目标 <0.05s）。中心节点（Node 1/3/6）最差，这些节点连接数最多（4个邻居），时隙争抢最激烈。

**3. Tx 成功率未达标**

59.3% < 目标 70%。但相比历史（50.9%），RL 已使成功率提升 8.4pp，控制冲突减少 29%，说明 RL 策略对吞吐量有正向作用。

**4. RL 策略从崩溃后 checkpoint 恢复**

本次加载的 `tdma_ppo_latest.pt` 对应 frame ~7000（崩溃后），avg_r 仅 +43（历史高峰 +82~+96）。

### 改进方案（2026-04-11 实施）

1. **从崩溃前 checkpoint 重训**：使用 `checkpoints/tdma_ppo_frame4000.pt`（崩溃前高峰期）
2. **增强公平性激励**：`r_gamma` 0.3 → 0.5，使策略更积极地追求均匀时隙分配
3. **防止熵坍缩**：`ent_coef` 0.05 → 0.1，保持更长时间的策略多样性
4. **减少过拟合**：`ppo_epochs` 4 → 2，每次更新梯度步数减半
5. **延长仿真**：`sim-time-limit` 20s → 100s（约 370 帧，11 次 PPO 更新），获得足够训练信号

启动命令：
```bash
./scripts/run_joint.sh --num_slots 10 --num_nodes 9 \
    --load_ckpt checkpoints/tdma_ppo_frame4000.pt \
    --ent_coef 0.1 --ppo_epochs 2 --r_gamma 0.5
```

## 问题 7：RL 从零学习上限分析（2026-04-11 系统实验）

**发现时间**：2026-04-11 多轮实验后综合结论

### 核心发现

经过大量实验，确认：**当前架构的 RL 无法从随机初始化提升至历史 +82~+96 水平**。

#### 数据对比

| 策略来源 | avg_r | 说明 |
|:--|:--|:--|
| C++ 启发式策略 | +82~+96 | 历史"高峰期"实际为启发式输出（ActionSender fix 前 3072 帧） |
| RL 从零学习（任何 ent_coef 配置） | +44~+52 | 与随机策略基线（+49）无显著差异，1760+ 帧无进展 |
| 随机策略基线 | +49 | frame 32 时即可观测，ppo_epochs=2/4 下保持平稳 |

#### 根本原因：共享策略协同问题

所有 9 个节点**共享同一组参数**，加上节点 ID one-hot 嵌入。从随机初始化出发：
1. 所有节点策略完全相同，倾向于选择**同一批时隙** → 大量冲突
2. Advantage 估计噪声大（动作随机，奖励信号弱）
3. 节点 ID one-hot 在随机权重下无法有效区分节点行为，需要协同梯度才能学出分工
4. 当 entropy 下降时，策略收敛到某个随机方向（不一定是协调方向），avg_r 不升反降

#### 历史 +82~+96 的真相（重要修正）

2026-03-29 记录的 "前 4800 帧 avg_r 维持 +82~+96" 中：
- **前 3072 帧**：C++ 使用启发式策略，RL 训练在启发式数据上进行（off-policy 预热）
- **第 3072 帧**：ActionSender 连接成功，RL 开始控制时隙选择
- **3072-4800 帧**：RL 从启发式"热身"后的良好初始点出发，短暂维持高性能后崩溃

这一现象是 Bug（ActionSender 延迟连接），而非 RL 的真实学习能力。ActionSender fix 后，此路径已不可复现。

### 崩溃阈值规律

| 实验 | 崩溃时 entropy | ent_coef |
|:--|:--|:--|
| 2026-03-15 基线 | 0.30 | 0.01 |
| Phase 1b（ppo_epochs=4, ent_coef=0.05） | 0.34 | 0.05 |
| frame7000+自适应（ent_coef=0.10） | 0.22 | 0.10 |

**结论**：崩溃阈值随 ent_coef 下移（更强熵正则可推迟崩溃），但并非"防止崩溃"，只是减慢了熵下降速度。当前实现无法真正稳定 entropy > 0.30。

## 问题 8：行为克隆（BC）预训练实验（2026-04-12）

### 8.1 独立 Agent 架构实验结论（问题 7 延续）

| 运行 | 帧数 | 初始 entropy | 最终 entropy | avg_r 范围 | 结论 |
|:--|:--|:--|:--|:--|:--|
| 独立 Agent Run 1（从零） | 1760 | 0.692 | 0.649 | +44~+52 | 无进展 |
| 独立 Agent Run 2（自适应 ent） | 1760 | 0.649 | 0.472 | +44~+52 | 无进展 |
| 独立 Agent Run 3（ent_coef=0.10） | ~1500 | ~0.472 | ~0.360 | +44~+52 | 无进展 |
| 独立 Agent Run 4（ent_coef=0.10） | 1440 | 0.359 | 0.250 | +44~+52 | 无进展 |

**结论**：独立 Agent + 自适应 ent_coef 在 ~6500 帧后仍无法突破 +49 随机基线。根本原因与共享 Agent 相同：单节点 32 样本/更新，梯度噪声过大，无法学出协调分工。

### 8.2 BC 预训练实现

**设计**：
- Python 连接状态管道（读状态），不发送动作 → C++ 自动使用启发式策略
- BC 训练对：`(feat_seq_{t-1}, Pt1_t)` 其中 Pt1_t = 启发式在帧 t-1 的申请概率
- Loss：`MSE(network_output, Pt1)`，lr=1e-3

**BC 训练曲线**（1088 帧，34 次更新）：

```
frame=32  BC_loss=0.1860   ← 初始（全 0 初始化 vs 启发式平均概率）
frame=320 BC_loss=0.1444   ← 快速下降阶段
frame=640 BC_loss=0.0801   ← 继续下降
frame=960 BC_loss=0.0578   ← 趋于收敛
frame=1088 BC_loss=0.0658  ← 平台期（~0.055~0.07）
```

**BC 后策略状态**：entropy ≈ 0.43（适度确定性），avg_r ≈ +13（低于随机基线 +49）

### 8.3 BC 效果分析：为何 avg_r = +13 低于随机基线

**启发式高性能（+82~+96）的真正来源**：

C++ 启发式的核心机制是 `avoidSlotsNextSchedule`：当某时隙上次发生冲突，下次自动降低该时隙申请概率（一次性退避）。这是一种**在线自适应碰撞回避**机制。

BC 训练目标 `Pt1` 是"特定上下文下启发式决策的平均"，缺少了这个自适应部分：

| 机制 | 启发式策略 | BC-RL 策略 |
|:--|:--|:--|
| 基础时隙偏好 | 有（基于队列长度、邻居状态） | 有（BC 学到平均偏好） |
| 碰撞自适应退避 | **有**（avoidSlotsNextSchedule） | **无**（输出固定概率，不响应上一帧碰撞） |

结果：BC 策略在争用时隙上反复请求，导致**持续性碰撞** → avg_r < 随机基线。

### 8.4 BC + RL 微调实验

| 参数组合 | 初始 avg_r | 最终 avg_r（14 更新后） | entropy 趋势 | 结论 |
|:--|:--|:--|:--|:--|
| ent_coef=0.10（第一次，帧对齐 bug） | +21 | -157 | 0.43→0.53 ↑ | 崩溃 |
| ent_coef=0.01（修复对齐后） | +13 | -176 | 0.43→0.49 ↑ | 崩溃 |

**崩溃模式分析**：
1. BC 策略持续请求争用时隙 → 大量冲突 → 负奖励
2. RL 梯度（A < 0）推动策略远离这些动作 → entropy 上升（更随机）
3. 随着 entropy 上升，策略变成"random RL"（avg_r ≈ +49）→ 但过程中经过 avg_r < 0 的谷底
4. 最终仍然收敛到 +49 随机基线（14 更新不够到达）

**对帧对齐 bug 的修正**：
```python
# Bug（原始）：feat_seq_prev 是添加当前 feat 之前的序列 → 2帧错位
feat_seq_prev = stack(window_before_append)  ← 使用时对应 state_{t-2}

# Fix：先 append，再构建当前帧序列
window.append(feat)  # 先加入
feat_seq_curr = stack(window_after_append)   ← 使用时对应 state_{t-1} ✓
```

### 8.5 根本限制总结

| 现象 | 根本原因 |
|:--|:--|
| RL 无法超越 +49 随机基线 | 样本效率不足（32 样本/节点/更新），梯度太噪，无法学出协调 |
| BC 启动策略低于随机基线 | BC 学到的是启发式"平均动作"，缺少 avoidSlotsNextSchedule 自适应退避 |
| BC + RL 微调崩溃 | BC 初始策略差（+13），RL 梯度需推翻 BC 结构才能改善，过程产生极大负奖励 |

### 8.6 下一步：RL 与启发式协同方案（推荐）

**核心思路**：让 RL 学习**调整**启发式，而不是**替代**启发式。

在 C++ 中修改 `scheduleRequests()`：
```cpp
// 当前：直接用 RL 概率替代启发式
float p = useRlAction ? getRlActionProb(slot) : heuristicPriority(slot);

// 建议：RL 输出乘数，叠加在启发式概率上
float heuristic_p = computeHeuristicPriority(slot);  // 含 avoidSlotsNextSchedule
float rl_factor   = getRlActionFactor(slot);          // RL 输出 [0.5, 2.0]，中性值=1.0
float p = clamp(heuristic_p * rl_factor, 0.0, 1.0);
```

好处：
- 启发式的自适应退避机制**始终生效**
- RL 只负责"调高/调低特定节点对特定时隙的偏好"（细粒度调优）
- 从中性初始化（rl_factor=1.0）出发，RL 有可能改善启发式

此方案需要 C++ 修改，预计工作量 2~3 天。

## 问题 9 — RL 乘数模式（Multiplier Mode）

**发现时间**：2026-04-12

**问题描述**：

历次 RL 实验（直接替换概率、独立 Agent、BC 预训练）均无法突破随机基线 +49，而启发式稳定 +82~+96。根本原因诊断：

- **RL 接口直接替换申请概率**：`reqProb = rlProb`（C++ `scheduleRequests()`）
- **`avoidSlotsNextSchedule` 退避机制通过 `slotOrder` 生效**，但 RL 可以对退避时隙赋予高概率，完全绕过退避效果
- **BC 预训练**学到"平均启发式"，输出 0.6~0.9 概率，同样绕过退避 → avg_r ≈ +13（低于随机基线 +49）

**解决方案：RL 乘数模式**

**已实施（2026-04-12）**，改动文件：

| 文件 | 改动 |
|:---|:---|
| `DynamicTDMA.cc` `scheduleRequests()` | 始终计算 `heurProb`，RL 输出 α 作为乘数因子 |
| `DynamicTDMA.h` | 新增 `myHeurProbs` 成员变量 + `RlFrameFeatures.heurProbs` 字段 |
| `rl/rl_agent.py` | `actor_head` 零初始化 + 特征追加 HeurProb（M维）+ `input_dim: 4M→5M` |
| `rl/rl_receiver.py` | `NodeObservation` 新增 `HeurProb` 字段 |
| `rl/ppo_trainer.py` | BC 预训练废弃（`--bc_frames` 抛 ValueError） |

**核心逻辑（C++ `scheduleRequests()`）**：

```cpp
// 旧版（退避失效）
reqProb = rlProb;  // 直接替换，绕过启发式

// 新版（乘数模式，退避保留）
double heurProb = max(dynamicPrio, minProb) + highLoadBoost;
myHeurProbs[slot] = heurProb;
if (getRlActionProb(slot, rlAlpha)) {
    reqProb = clamp(heurProb * rlAlpha * 2.0, 0.0, 1.0);
    // rlAlpha≈0.5 → 乘数=1.0，等效纯启发式
    // rlAlpha→1.0 → 乘数=2.0，放大（最大为2×heurProb）
    // rlAlpha→0.0 → 乘数≈0，抑制该时隙
}
```

**乘数上限分析（max = min(1.0, 2×heurProb)）**：

| 优先级 | heurProb 最小值 | RL 最大 reqProb | 说明 |
|:---|:---|:---|:---|
| 高优先级 | 0.6 | min(1.0, 1.2) = 1.0 | RL 可强制全满 |
| 低优先级 | 0.2 | min(1.0, 0.4) = 0.4 | 保留优先级排序 |

结论：满足优化需求。若后续需更强低优先级扩展能力，可将系数 2.0 改为 3.0。

**网络初始化改动**：

```python
# rl_agent.py LSTMActorCritic.__init__()
self.actor_head = nn.Linear(lstm2_hidden, num_slots)
nn.init.zeros_(self.actor_head.weight)   # 新增
nn.init.zeros_(self.actor_head.bias)     # 新增
# Sigmoid(0) = 0.5 → 初始乘数 = 1.0，训练起点等价纯启发式
```

**状态向量变化**：

```
旧版: Bown(M) + T2hop(2M) + scalar(10) + Pt1(M) + NodeID(N) = 4M+10+N = 59维
新版: Bown(M) + T2hop(2M) + scalar(10) + Pt1(M) + HeurProb(M) + NodeID(N) = 5M+10+N = 69维
```

新增 `HeurProb`（本帧启发式概率向量）让 RL 感知乘数基准，便于学习精确调整策略。

**注意**：新状态维度与旧 checkpoint（59维）不兼容，需从零开始训练。

**实验结果（2026-04-12 验证）**：

| 指标 | 纯启发式基线 | RL 直接替换 | BC 预训练 | RL 乘数模式（实测） |
|:---|:---|:---|:---|:---|
| avg_r 初始值 | +44~+49 | +44~+52 | +13 | +44（α=0.5，等效启发式） |
| avg_r 稳定值 | +44~+49 | +44~+52 | +13 | +44（1760帧无提升） |

**基线修正**：文档曾声称纯启发式基线 avg_r ≈ +82~+96，经验证为旧代码（`494c4e4` 项目重组前）的产物，当前代码下纯启发式基线为 **+44~+49**。历史 +82~+96 因 DynamicTDMA.cc 的 Bug 修复和同步机制变更已不可复��。所有保存的日志（2026-04-11~12）均证实从零开始的 avg_r 在 +42~+49 范围。

## 改进 5：网络瘦身 + 训练节奏 + 奖励重塑（三合一）

**实施时间**：2026-04-12

**问题诊断**：乘数模式实施后，RL 训练 1760 帧仍无法超越启发式基线（avg_r ≈ +44）。根因分析：

1. **网络严重过参数化**：旧版 LSTM-1(128) + LSTM-2a(64) + LSTM-2c(64) 共 ~192K 参数，但每次 PPO 更新仅 32 帧样本（9节点 × 32帧 = 288 条数据），参数/样本比 > 600:1，梯度被噪声完全淹没
2. **更新间隔过短**：`update_every=32` 导致每次 PPO 更新的轨迹过短，GAE 估计方差极大
3. **奖励信号弱**：原始 avg_r ≈ +44 波动范围 ±10，PPO 无法区分好坏动作

### 改动 A：网络瘦身（`rl/rl_agent.py`）

去掉 LSTM-2 层，LSTM-1 hidden 从 128 缩至 32：

```
旧版（~192K 参数）:
  输入(69) → LSTM-1(128) → Actor: LSTM-2a(64) → Linear(10) → Sigmoid
                          → Critic: LSTM-2c(64) → Linear(1)

新版（~13.5K 参数）:
  输入(69) → LSTM-1(32) → Actor: Linear(10) → Sigmoid（零初始化）
                         → Critic: Linear(1)
```

参数量对比：
- 旧版：LSTM-1(4×128×(69+128+1))=101K + LSTM-2a(4×64×(128+64+1))=49K + LSTM-2c(49K) + 头部 ≈ 192K
- 新版：LSTM-1(4×32×(69+32+1))=13K + actor_head(32×10)=320 + critic_head(32×1)=32 ≈ 13.5K

**关键保留**：actor_head 零初始化（Sigmoid(0)=0.5 → 乘数=1.0 → 等效纯启发式起点）。

### 改动 B：增大更新间隔（`rl/ppo_trainer.py`）

`update_every` 从 32 → 128。每次 PPO 更新的样本量从 288 增至 1152（9节点 × 128帧），梯度估计更稳定。

### 改动 C：差分奖励重塑（`rl/ppo_trainer.py`）

每节点维护 EWMA 奖励基线（衰减系数 0.95），存入 buffer 的奖励为差分值：

```python
raw_r = compute_reward(obs)
baseline = reward_baselines[nid]
reward_baselines[nid] = baseline + 0.05 * (raw_r - baseline)
shaped_r = raw_r - baseline  # 存入 buffer
```

效果：消除奖励绝对值的干扰，策略梯度只关注"比历史平均好还是差"，减小方差。日志 `avg_r` 仍用原始奖励便于历史对比。

### 首次实验结果（2026-04-12）


### 改进 5.1：二值动作 + 奖励延迟对齐

**实施时间**：2026-04-12（紧接改进 5）

**发现的两个架构级 Bug**：

#### Bug 1：动作-环境断开

原代码将网络输出的**连续 α ≈ 0.5 发送给 C++**，而 PPO 优化的是**二值采样动作**。C++ 始终收到 α≈0.5 → 乘数=1.0 → reqProb=heurProb，环境行为恒定不变。PPO 的 advantage 对 a=0 和 a=1 相同，梯度正负抵消 → L_actor ≈ 0。

**修复**：改为发送 Bernoulli 采样的二值动作 `node_actions`（0或1），C++ 乘数模式下 action=1 → 乘数=2.0（强申请），action=0 → 乘数=0.0（不申请）。

#### Bug 2：奖励时序错位

帧 t 的 Nsucc/Ncoll 反映帧 t-1 的调度结果（因 C++ `writeRlFeatures()` 在数据阶段末尾调用，Nsucc 是本帧已完成传输的统计量，由帧 t 的 `scheduleRequests()` 决定，而该调度使用的 RL 动作来自帧 t-1 的 `readRlActions()`）。

原代码将帧 t 的奖励 r_t 和帧 t 的动作 a_t 配对，但 r_t 实际是 a_{t-1} 的结果 → advantage 不反映 a_t 的质量 → 梯度是噪声。

**修复**：引入 `pending_transitions` 字典，帧 t 的 transition 在帧 t+1 到来时才用 r_{t+1} 完成，确保 (s_t, a_t, r_{t+1}) 因果对齐。

#### 实验对比（83 次 PPO 更新）

| 实验 | 修复内容 | entropy 变化 | L_actor |
|:---|:---|:---|:---|
| 发连续 α | 无 | 0.6931→0.6923（Δ=-0.0008） | ≈ 0 |
| 二值动作 | Bug 1 | 0.6931→0.6927*（28次更新） | ≈ 0 |
| 二值+延迟对齐 | Bug 1+2 | 0.6931→0.6915（**Δ=-0.0016**） | ≈ 0 但有微弱趋势 |

修复后 entropy 下降速度翻倍，但仍然极慢。L_critic 持续下降（1.0→0.55），Critic 学习正常。

#### 残余问题：多智能体信用分配

9 节点独立学习，每节点的 Nsucc/Ncoll 同时受自身动作和其他 8 个节点的随机动作影响。单节点动作的边际效应被多智能体噪声淹没，策略梯度信噪比极低。这是独立 Agent + 局部观测约束下的固有困难。

**可能的后续方向**：
1. **增大 update_every 至 512+**：更多样本平均掉多智能体噪声
2. **差分奖励 baseline 使用 Critic 的 V(s)**：用 GAE 已有的 advantage 估计代替简单 EWMA，更好地减小方差
3. **渐进式探索**：初期保持 α≈0.5（少偏离启发式），缩小探索范围降低其他节点噪声的影响
4. **增大学习率**：小梯度需要大步长来穿过噪声（但有发散风险）

## 改进 6：逐时隙奖励分解（Per-Slot Reward Decomposition）

**日期**：2026-04-13

### 问题诊断

改进 5.1 后 entropy 下降速度仍极慢（0.0016/83 次更新），根因分析发现**聚合奖励函数是核心瓶颈**：

1. **信用分配缺失**：每节点每帧做 M=10 个独立二值动作，但只收到 1 个标量奖励（Nsucc - Ncoll + Jlocal - Wt）。PPO 无法区分哪个时隙的决定好、哪个坏。
2. **Jlocal/Wt 增加噪声**：Jlocal 是全网聚合指标，Wt 受流量随机性影响大，对单帧时隙选择几乎无有效梯度信号。
3. **多智能体噪声被聚合放大**：单个时隙动作的边际贡献 ≈±1，但奖励总方差来自 10 个时隙叠加 × 9 节点噪声。

**信噪比估算**：单时隙动作对聚合奖励的信噪比 ≈ 1/(√10 × 多节点因子) ≈ 1/30，PPO 梯度被噪声淹没。

### 改进方案

**方向 A：逐时隙奖励分解**

#### C++ 端改动

`writeRlFeatures()` 新增 `SlotResult` 字段（M 位字符串），逐时隙标记：
- `'0'` = 未申请（myPriorities[s] ≤ 0）
- `'1'` = 申请且成功（mySlots[s] == true）
- `'2'` = 申请但失败（mySlots[s] == false）

```json
"reward_signal": {"Nsucc":3, "Ncoll":1, "SlotResult":"0012001010", ...}
```

#### Python 端改动

1. **逐时隙奖励**（`compute_per_slot_reward`）：
   ```python
   r_s = +1.0  if SlotResult[s] == '1'  # 成功
   r_s = -1.0  if SlotResult[s] == '2'  # 失败
   r_s =  0.0  if SlotResult[s] == '0'  # 未申请
   ```

2. **逐时隙 Critic**：`critic_head` 从 `Linear(32, 1)` → `Linear(32, M)`，每个时隙独立价值估计 V_s(t)。

3. **逐时隙 PPO**：
   - ratio 不再对 M 维 log_prob 求和，改为逐时隙 `ratio_s = exp(log_prob_s - log_prob_old_s)`
   - advantage 为 (N, M) 形状，每时隙独立 GAE
   - Clipped surrogate 逐时隙计算后取均值

4. **逐时隙 EWMA 基线**：`reward_baselines[nid]` 从标量改为 (M,) 向量，差分奖励逐时隙独立。

### 分析

1. **逐时隙信用分配有效**：每个 Bernoulli 策略独立接收该时隙的成功/失败反馈，梯度信噪比从 ~1/30 提升至 ~1/1，entropy 收敛提速 100x+。

2. **avg_r 为负是正常现象**：逐时隙奖励的 avg_r = Σ(+1 成功, -1 失败) 累积 128 帧 × 9 节点平均。二值动作模式下，action=0 → 乘数=0（完全屏蔽该时隙），action=1 → 乘数=2（加倍申请）。随机策略（entropy≈0.693）约一半时隙被屏蔽、一半加倍申请，比纯启发式差。但 avg_r 从 -302 上升到 -244，趋势正确。

3. **avg_r 量级与旧版不可直接比较**：旧版 avg_r = Nsucc - Ncoll + 0.3*Jlocal - 0.2*Wt ≈ +44~+49，新版 avg_r = Σ(逐时隙 ±1) × 128 帧 / 9 节点，量级和含义不同。

### 兼容性说明

- Critic 从 1 输出改为 M 输出，旧 checkpoint 不兼容，需从零训练
- C++ 端 JSON 新增 `SlotResult` 字段，Python 端做了兼容处理（缺失时默认空字符串）
