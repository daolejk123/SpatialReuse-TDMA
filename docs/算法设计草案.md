## 一、设计结论

当前 `B` 已经不是“纯 PPO”，而是一个：

- 以启发式申请概率为基准；
- 以 PPO/LSTM 输出乘数进行在线修正；
- 以逐时隙 Bernoulli 动作为执行形式；
- 叠加 two-hop 安全先验、service debt、request budget 和若干情境门控；
- 面向动态 TDMA / STDMA 空间复用的混合调度器。

现有实验说明，继续增加单独的 reward patch 或场景化 gate，已经进入收益递减阶段：

- `two-hop mask` 能改善稀疏 UAV 场景的安全性和吞吐，但不能单独解决饥饿；
- `service debt` 能降低 starvation，但若只靠固定 boost，容易牺牲 pedestrian 场景吞吐；
- `wt_threshold` 和 `context_gate` 都表现出明显场景依赖，说明“用手工阈值替代全局目标”并不稳。

因此，下一版最合理的升级方向不是继续补丁式加规则，而是把“长期未服务”正式提升为优化对象：

`B_constrained_debt = 启发式先验 + LSTM PPO + two-hop mask + 显式服务约束 + 状态化 debt + 受控申请预算`

它的目标不是追求每个场景都立刻超过 `B`，而是先建立一个更干净、更可解释、能继续扩展到 tail-risk 和 multi-agent CTDE 的主干。

## 二、应用场景与问题定义

### 2.1 场景

面向 MANET / UAV-inspired 动态拓扑下的分布式 TDMA 接入：

- 每个节点只掌握本地观测和邻域摘要；
- 同一数据时隙允许空间复用；
- 一跳 / 两跳冲突会导致申请失败或资源浪费；
- 拓扑、队列和业务负载随时间变化；
- 评价目标同时包含吞吐、排队稳定性、公平性和尾部等待。

### 2.2 当前决策形式

对节点 `i`、时隙 `s`、帧 `t`：

- 启发式模块给出基准概率 `h_i,s(t)`；
- RL actor 输出乘数变量 `alpha_i,s(t)`；
- 当前系统按类似 `p_i,s(t) = f(h_i,s(t), alpha_i,s(t))` 的方式得到最终申请概率；
- 每个时隙独立采样 `a_i,s(t) in {0, 1}`；
- 仿真器返回逐时隙成功、失败、冲突和队列变化。

### 2.3 设计目标

主目标按优先级排序：

1. 在不破坏现有 `B` 主线的基础上，显式降低长期未服务和尾部等待。
2. 保持稀疏 UAV 场景中已验证有效的 two-hop safety prior。
3. 降低对固定手工阈值和场景特定 gate 的依赖。
4. 给后续 `CVaR` 风险目标和 `MAPPO / centralized critic` 留出接口。

### 2.4 非目标

第一版不做：

- 不直接替换主表中的 `B`；
- 不立刻把 actor 改成 GNN；
- 不立刻改成 MADDPG / QMIX / value-based 主干；
- 不追求一次性解决所有场景 trade-off；
- 不把物理层抽象扩展到真实 fading / SINR。

## 三、现有结构诊断

### 3.1 当前结构中应保留的部分

| 模块 | 是否保留 | 原因 |
|---|---:|---|
| 启发式申请概率 | 保留 | 已提供可用先验，RL 不必从零学会基本接入行为。 |
| RL 乘数融合 | 保留 | 比“全量替代启发式”更适合在线小样本训练。 |
| LSTM | 保留 | 当前环境部分可观测，历史信息本身有价值。 |
| 逐时隙 critic / reward | 保留 | 与动作粒度一致，credit assignment 比整帧 reward 更细。 |
| two-hop mask | 保留 | 对稀疏空间复用场景已证明有意义，且符合冲突图先验。 |
| request budget | 保留 | 能抑制“所有安全槽都申请”的过激行为。 |

### 3.2 当前结构的主要瓶颈

| 问题 | 当前表现 | 根因 |
|---|---|---|
| 公平性靠 reward patch 维持 | v4、`B_masked_debt*` 均出现吞吐与 fairness 的场景 trade-off | 长期约束被压扁进瞬时 reward，目标耦合不清。 |
| debt 更多是动作修饰器 | 通过 boost 和成功 reward 触发，仍依赖阈值 | debt 没有成为统一的优化状态。 |
| gate 逐渐变多 | density、Wt、context 等多套规则并存 | 缺少一个能解释“何时该让步”的上层目标。 |
| 独立 PPO 面对耦合环境 | 节点之间互相改变对方的数据分布 | 典型 multi-agent non-stationarity。 |
| 只优化均值更容易忽视尾部 | `Wt_P95`、starvation 往往需要额外修补 | 平均 reward 不能代表最差节点或最差时段。 |

## 四、核心设计

### 4.1 总体架构

```text
局部观测 O_i(t)
  ├─ 当前状态: Bown / T2hop / Cctrl / Qt / Wt / Share / Jain / ...
  ├─ 历史状态: LSTM window
  ├─ 启发式先验: HeurProb
  └─ 新增约束状态: debt_i(t), dual_i(t), recent_service_gap_i(t)
          ↓
LSTM Actor-Critic
          ↓
alpha_i,s(t)
          ↓
启发式概率融合
          ↓
two-hop safety mask
          ↓
受控预算投影 / 归一化
          ↓
逐时隙 Bernoulli 采样
          ↓
仿真反馈: success / collision / queue / waiting
          ↓
reward + cost + debt update + PPO update
```

### 4.2 状态设计

在现有 [rl/rl_agent.py](../rl/rl_agent.py) 状态上，新增三类字段：

| 字段 | 含义 | 作用 |
|---|---|---|
| `service_gap` | 连续未成功服务帧数，或其归一化值 | 告诉策略“谁已经被饿太久”。 |
| `debt_level` | 相对服务欠账强度 | 作为长期公平压力的直接状态。 |
| `dual_pressure` | 当前约束拉格朗日乘子或 virtual queue 强度 | 让 actor 感知系统当前对公平约束的重视程度。 |

可选第二阶段字段：

| 字段 | 含义 | 作用 |
|---|---|---|
| `tail_wait_risk` | 最近窗口内高等待分位风险 | 为 `CVaR` 版本服务。 |
| `recent_success_rate` | 近期申请成功率 EWMA | 作为动态预算的连续反馈。 |
| `queue_growth_rate` | 队列增长趋势 | 区分“真的需要抢资源”与“只是偶发未服务”。 |

设计原则：

1. 能由现有在线闭环稳定得到的量，优先显式入状态；
2. 能通过状态学习解决的，不再继续新增硬编码 gate；
3. 阈值仍可保留作 warm start，但不应成为长期主机制。

### 4.3 动作设计

第一版继续使用现有动作形式，不改动作空间：

```text
base_prob_i,s = heuristic_prob_i,s
raw_prob_i,s  = fuse(base_prob_i,s, alpha_i,s)
safe_prob_i,s = raw_prob_i,s * mask_i,s
final_prob_i,s = BudgetProject(safe_prob_i, debt_level_i, budget_i)
```

### 设计理由

- 继续保留启发式基准，能降低训练难度；
- 继续使用 `two-hop mask`，能把明确非法动作从搜索空间里删掉；
- `BudgetProject` 替代“所有安全槽统一 boost”，能避免 fairness 目标把系统推向过申请；
- 第一版不改变 `Bernoulli per-slot` 形式，能最大限度复用已有实验链路。

### 推荐预算机制

令每节点期望申请量满足：

```text
sum_s final_prob_i,s <= base_budget_i + debt_budget_bonus_i
```

其中：

- `base_budget_i` 由常规负载和安全槽情况决定；
- `debt_budget_bonus_i` 随 `debt_level_i` 增大，但设置硬上限；
- 若成功率下降或队列并未持续增长，则预算回落。

这比当前“先 boost 再全局缩放”的写法更适合作为正式算法叙事。

### 4.4 奖励与约束

#### 4.4.1 主 reward

主 reward 保持面向效率和稳定性：

```text
r_t =
  + w_tx * successful_tx
  - w_coll * collision
  - w_queue * queue_pressure
  - w_idle * idle_with_queue
```

设计意图：

- throughput / spatial reuse 仍然是主收益；
- collision 和 queue_pressure 继续约束系统效率；
- `idle_with_queue` 只保留轻量惩罚，不再承担长期公平主责。

#### 4.4.2 约束 cost

第一版建议至少建模两类 cost：

| 约束 | 示例定义 | 目的 |
|---|---|---|
| 服务频率约束 | `c_service_i(t) = 1[queue_i>0 and frames_since_tx_i > tau_service]` | 防止节点长期得不到服务。 |
| 饥饿约束 | `c_starve_i(t) = max(0, frames_since_tx_i - tau_starve)` | 抑制严重长尾未服务。 |

可选第二阶段加入：

| 约束 | 示例定义 | 目的 |
|---|---|---|
| 尾部等待约束 | `c_wait_i(t) = max(0, Wt_i - tau_wait)` | 直接对应 `Wt_P95` / tail latency。 |
| 申请预算约束 | `c_req_i(t) = max(0, sum_s p_i,s - budget_i)` | 避免公平目标诱发过申请。 |

#### 4.4.3 训练目标

采用 Lagrangian 形式表达：

```text
maximize   E[sum_t gamma^t r_t]
subject to E[sum_t gamma^t c_k(t)] <= d_k
```

等价训练时可写成：

```text
L_total = L_PPO_reward - sum_k lambda_k * L_cost_k
lambda_k <- [lambda_k + eta_lambda * (J_cost_k - d_k)]+
```

第一版实现上不必一步到位复刻 ACPO。更务实的落地方式是：

1. 保留现有 PPO 更新；
2. 新增 cost trajectory、cost advantage、cost return；
3. 增加一组在线更新的 `lambda_k`；
4. actor loss 中加入 cost advantage 项；
5. 先验证该机制是否比固定 `starvation_penalty_coef` 更稳。

### 4.5 debt 更新

推荐把 debt 设计成显式状态，而非只做临时 boost：

```text
debt_i(t+1) =
  clip(
    debt_i(t)
    + arrival_pressure_i(t)
    + unmet_service_i(t)
    - served_i(t),
    0,
    debt_max
  )
```

可用的第一版近似：

```text
debt_i(t+1) =
  clip(
    debt_i(t)
    + 1[queue_i(t) > 0 and not served_i(t)]
    - rho * 1[served_i(t)],
    0,
    debt_max
  )
```

设计理由：

- debt 对“连续未服务”有记忆；
- served 后自然偿还，不需要依赖下一帧 `Wt` 是否刚好越阈；
- debt 本身进入状态后，actor 可以学到“何时该让位，何时不该抢”。

## 五、为什么这套结构更可能优于当前版本

### 5.1 比继续堆 reward patch 更稳

当前很多 trade-off 来自把多个目标揉进一个瞬时 reward：

- 高吞吐；
- 低冲突；
- 低队列；
- 低等待；
- 低 starvation。

这些目标在不同场景下并不总同向。约束化后：

- `reward` 负责效率；
- `cost` 负责边界；
- `lambda` 自适应调节当前该为 fairness 付出多少代价。

这比手工把 `starvation_penalty_coef` 从 `0.05` 调到 `0.1` 更能解释，也更有机会跨场景泛化。

### 5.2 比固定 gate 更能处理动态场景

固定 gate 的问题不是“规则完全错”，而是它只能回答一个局部问题：

- `Wt` 高不高；
- density 稀不稀；
- topology 是否变了。

但调度真正要解决的是全局权衡：

- 现在多申请一个 slot，是否值得牺牲别的节点机会；
- 当前是暂时波动，还是结构性服务不足；
- 此时应该压制碰撞，还是优先偿还欠账。

把 debt 和 dual pressure 放入状态后，policy 至少有机会基于历史和当前负载学这个权衡，而不是被单个阈值强行切换模式。

### 5.3 比纯均值优化更适合你的指标

你的论文指标并不只看均值：

- `Wt_P95`
- starvation
- `queue_stability_slope`
- Jain fairness

这些本质上都是长尾、稳定性或弱节点指标。  
因此，先做 constrained debt，再扩展到 `CVaR / tail-risk`，比继续围绕平均 goodput 做局部修补更符合你的评价体系。

### 5.4 比立即上 GNN / MAPPO 更稳妥

你当前最大的已知问题不是“表示能力不够”，而是：

- fairness 目标没有被正式建模；
- 多个 gate 的职责混乱；
- 现有结构还没有被约束版充分验证。

所以合理顺序是：

1. 先把目标函数理顺；
2. 再把训练结构升级到 centralized critic；
3. 最后再在需要跨规模泛化时引入图表示。

## 六、算法流程

```text
for each frame t:
  receive local observations for all nodes

  for each node i:
    build feature sequence:
      current local state
      + heuristic prior
      + debt_i
      + dual pressure

    alpha_i, value_i = actor_critic(feature_seq_i)
    raw_prob_i = fuse(heuristic_prob_i, alpha_i)
    masked_prob_i = apply_twohop_mask(raw_prob_i)
    final_prob_i = project_request_budget(masked_prob_i, debt_i)
    sample per-slot Bernoulli action a_i

  send actions to simulator
  receive outcome:
    success / collision / queue / waiting / fairness

  for each node i:
    compute reward_i
    compute constraint costs_i
    update debt_i
    store transition

  every K frames:
    compute reward advantages
    compute cost advantages
    update PPO actor/critic
    update lambda for each constraint
```

## 七、建议的版本路线

### 7.1 `B_constrained_debt`

第一优先级，建议立即实现。

特征：

- 现有 `B` 主干；
- two-hop mask；
- 预算投影；
- 显式 service / starvation cost；
- debt 和 dual pressure 进入状态；
- PPO-Lagrangian 近似更新。

### 7.2 `B_constrained_debt_cvar`

第二优先级，在前一版稳定后再做。

新增：

- tail waiting / tail starvation 的窗口采样；
- 基于高分位或 CVaR 近似的风险 cost；
- 对应 `Wt_P95`、最差节点等待、最差窗口稳定性。

### 7.3 `B_mappo_constrained`

第三优先级。

新增：

- centralized critic；
- actor 仍保持分布式执行；
- 训练时引入全局队列、全局公平性、全局拓扑摘要；
- 重点验证是否缓解独立 PPO 的非平稳性。

## 九、第一版参数草案

| 参数 | 建议初值 | 说明 |
|---|---:|---|
| `service_threshold` | `5` | 与当前 service debt 经验保持一致，便于可比。 |
| `starve_threshold` | `5` | 与现有 starvation 定义对齐。 |
| `debt_max` | `20` | 与当前 `service_debt_max_frames` 保持同量级。 |
| `lambda_lr` | `1e-3 ~ 1e-2` | 需要单独消融；过大会引起 fairness 过补偿。 |
| `service_cost_limit` | 由 v3 / v4 统计反推 | 不建议凭经验硬写，先用现有正式实验估计。 |
| `starve_cost_limit` | 由 sparse 主场景设定 | 目标应低于当前 `B`，但不能逼近零。 |
| `base_budget` | 继承现有值 | 先不引入额外自由度。 |
| `debt_budget_bonus_cap` | 小幅正值 | 只允许 debt 有界增加请求机会。 |

> 2026-05-17 连续分配复核：按上条建议实现 `B_constrained_smooth_budget`，让所有有债节点保留 `0.50` 最小份额，再按 debt 排名平滑增配。它比硬 top-k 温和，但 pilot 仍未优于 `budget_only`：pedestrian long-gap 总量 `2621 > 1584`，UAV sparse 则从 `0` 新增到 `458`，且 goodput/PDR 退到 `0.1395/0.0680`。说明问题不只在“硬切”本身，后续应转向更直接的还债动作或局部执行层反馈，而不是继续微调 rank-based bonus。
>
> 2026-05-17 显式还债动作复核：进一步实现 `B_constrained_repay_action`，每帧只为 top-1 高债节点强制保留一个安全槽位，并把该 override 从 actor 训练 mask 中剔除。结果说明“更直接”仍不等于“更有效”：pedestrian 虽把 starvation 压到 `0.5000`、queue_slope 降到 `11.1240`，但 goodput/PDR 退到 `0.2307/0.1127`，按 queued-frame 归一化后的 long-gap rate 反而升到 `5.84%`；UAV sparse 也没有超过 `budget_only` 的 tail 表现。后续若继续，不应再扩大 deterministic override，而应先定义同时受吞吐约束的局部 repayment 目标。
>
> 2026-05-17 局部可学习 repayment 复核：先后实现 `B_constrained_repay_reward` 与 `B_constrained_repay_target`。前者只在成功发送后给 top-tail 节点局部 bonus，但两轮 smoke 的 `repay_bonus` 都恒为 `0.000`，说明 success-only 信号无法为长期失服节点 bootstrap。后者改成 same-frame top-tail 的概率软目标，pilot 中虽改善部分均值指标，真实 tail 仍未修复：pedestrian long-gap rate `2.3266% > budget_only 2.0340%`，UAV sparse 也从 `0` 新增到 `31` 个 long-gap 帧。到这一步，继续围绕 per-node debt heuristic 做小修补的收益已经很低；更合理的后续方向应是 tail-risk / CVaR 或 CTDE / centralized critic，而不是再追加新的局部 bonus。
>
> 2026-05-17 CTDE 复核：进一步实现共享 actor 对照 `B_constrained_shared_actor` 与 centralized critic 版 `B_constrained_mappo`。`mappo` 的 actor 仍只看本地观测，critic 才读取全网状态序列。两场景 pilot 说明 centralized critic 确实比局部补丁更接近目标：pedestrian long-gap rate `1.1220%→0.9759%`，UAV sparse long-gap `1047→0`、starvation `0.5000→0.2500`；但它仍未通过本轮门槛，因为 queue_slope 分别从 `17.1714→18.2536`、`10.9904→12.4040`，都超过允许的 `5%` 回退。结论是“CTDE 值得保留为下一轮方向”，不是“当前版本已经可以升为正式方法”。
>
> 2026-05-17 风险 critic 复核：进一步实现 `B_constrained_mappo_risk`，让 centralized critic 显式读取逐节点 `tail_risk` 与 `queue_risk`，并补充帧级 mean/top-fraction 风险统计，actor 仍保持本地执行。pilot 说明这一步确实改变了 critic 的偏好：pedestrian long-gap rate 相比 `budget_only` 从 `3.3073%` 降到 `2.5124%`、最差 gap `3600→1837`；UAV sparse 中也清除了原版 `mappo` 新增的 `15` 个 long-gap 帧，并把 queue_slope `10.5221→8.6565`。但它仍未解决根问题：pedestrian goodput/PDR 退到 `0.2544/0.1243`，UAV sparse queue_slope 仍高于 `budget_only` 的 `6.3230`，starvation 也更差。结论是“显式风险特征有帮助，但单靠多喂特征还不够”；若继续，应转向 critic 目标或 advantage weighting，而不是继续堆局部 heuristic。
>
> 2026-05-17 联合价值目标复核：继续实现 `B_constrained_mappo_joint_value`，让共享 rollout 直接按 `reward - tail_coef * tail_risk - queue_coef * queue_risk` 计算 GAE 与 return。`0.5/0.5` pilot 说明“改目标”确实比“只加特征”更强：pedestrian long-gap rate `4.2272%→1.9607%`，UAV sparse 的 goodput/PDR、queue_slope、starvation 也同时改善到 `0.2756/0.1349`、`9.1623`、`0.2500`。但它仍没有过门槛，因为 UAV sparse 从原本 `0` 个 long-gap 帧新增到 `286` 个；把 tail 权重提到 `1.0/0.5` 的补充 smoke 也没有得到更好的开局。结论是“联合目标方向成立，但把 tail 与 queue 压成一个线性标量还不够”；下一步更适合多头 critic 或 tail-aware advantage gating，而不是继续只扫单一线性权重。
>
> 2026-05-17 多通道价值复核：继续实现 `B_constrained_mappo_multihead` 与 `B_constrained_mappo_tail_gate`。前者把 centralized critic 拆成 reward/tail/queue 三头，后者让 tail 压力在 actor advantage 中拥有不可被 queue 收益完全抵消的独立通道。pilot 结果把两者区分得很清楚：`multihead` 在 pedestrian 的 long-gap rate `1.6707%` 略低于同批 `budget_only 1.6932%`，在 UAV sparse 也把 `joint_value` 的 long-gap `592→58`，说明分头 credit assignment 确实更接近目标；但它仍以 pedestrian 的 goodput/PDR 与 queue 回退为代价。`tail_gate` 则在 UAV sparse 最强，long-gap 保持 `0`、queue_slope `4.8366`、starvation `0.3333`，可是在 pedestrian 反而升到 `2.6176%` long-gap。结论是“多头 critic 值得保留，第一版 gating 明显场景依赖”；下一步若继续，应先解决跨场景一致性，而不是直接把任一分支升为正式方法。

## 十、实验设计

### 10.1 必做对照

| 对照 | 目的 |
|---|---|
| `B` | 当前论文主方法。 |
| `B_masked_debt_budget` | 当前最接近的新版本。 |
| `B_constrained_debt` | 新方法。 |
| `greedy_stdma_2hop` | 强传统基线。 |

### 10.2 场景

至少保留：

- `N12_grid_manet_pedestrian`
- `N12_grid_manet_uav_sparse`

如果时间允许，再加：

- 一个动态扰动场景；
- 一个高负载 / 更密集场景。

### 10.3 主指标

| 指标 | 关注点 |
|---|---|
| `goodput_per_slot` | 效率 |
| `PDR` | 有效交付 |
| `queue_stability_slope` | 稳定性 |
| `Wt_P95` | 尾部等待 |
| `starvation` | 长期未服务 |
| `jain_tx_mean` / `jain_tail100` | 公平性 |
| `mac_efficiency` | 申请质量 |

### 10.4 关键消融

| 消融 | 验证问题 |
|---|---|
| `constraint off` | 显式约束是否真的比固定 reward 好。 |
| `debt state off` | debt 入状态是否优于只做动作 boost。 |
| `mask off` | two-hop safety prior 对新方法是否仍必要。 |
| `budget off` | fairness 改善是否只是通过过申请换来的。 |
| `dual off` | adaptive `lambda` 是否优于固定 penalty。 |

### 10.5 验收标准

`B_constrained_debt` 进入下一轮正式实验，应至少满足：

1. 在 UAV sparse 中，相比 `B`：
   - starvation 明显下降；
   - queue slope 不劣化；
   - goodput / PDR 不低于现有 `B_masked_debt_budget` 可接受区间。
2. 在 pedestrian 中，相比 `B_masked_debt_budget`：
   - throughput 回退幅度收敛；
   - starvation 仍保持改善；
   - 不出现 MAC efficiency 明显塌陷。
3. 相比固定 reward 版本：
   - 约束违约曲线更平滑；
   - 不再依赖新增场景特定 gate 才能工作。

## 十一、风险与应对

| 风险 | 表现 | 应对 |
|---|---|---|
| `lambda` 振荡 | fairness 与 throughput 来回摆动 | 降低 `lambda_lr`，使用滑动平均 cost，必要时双时间尺度更新。 |
| fairness 过补偿 | request 激增、MAC efficiency 下降 | 保留 budget projection，并监控申请保守度。 |
| sparse 有效、pedestrian 仍回退 | 与当前 debt 版本类似 | 检查 cost limit 是否一刀切，必要时做密度条件化约束上限，而不是继续加硬 gate。 |
| 训练不稳定 | PPO loss 抖动、熵塌陷 | 先只加一类 service cost，再逐步加入 starvation / wait cost。 |
| 论文叙事复杂化 | 方法像多个模块堆叠 | 主文只保留“heuristic prior + constrained debt”两层叙事，mask / budget 作为安全执行层。 |

## 十二、推荐实施顺序

1. 在 `rl/ppo_trainer.py` 中补齐：
   - `ConstraintState`
   - `CostTransition`
   - `lambda` 更新
   - actor loss 中的 cost advantage
2. 在 `rl/rl_agent.py` 中把 `debt_level`、`dual_pressure` 加入输入特征。
3. 新增 `B_constrained_debt` 方法入口，不覆盖现有 `B`。
4. 先跑 smoke，再跑单 seed pilot：
   - `pedestrian`
   - `uav_sparse`
5. 若 pilot 通过，再做 `1 2 3` seeds 小正式。
6. 小正式成立后，再决定：
   - 要不要加 `CVaR`
   - 要不要升到 centralized critic

## 十三、当前结论

这份草案建议的不是一次“大改”，而是一次主线重整：

- 保留现有已经验证有效的工程结构；
- 把 fairness 从“补丁”升格为“约束”；
- 把 service debt 从“动作小技巧”升格为“显式状态”；
- 用 `B_constrained_debt` 作为后续论文主创新的真正候选，而不是继续把所有新想法都堆到 `B_masked_debt_*` 系列里。

如果这一版成立，后续扩展顺序也会更清楚：

1. 先做 constrained debt；
2. 再做 tail-risk；
3. 最后再做 CTDE / graph representation。
