# 论文指标与性能对比基准

> 目的：把已核验论文中的评估指标、性能结论和本项目可采集指标对齐，作为后续 OMNeT++/PPO 仿真实验的对比口径。
> 关联文档：[项目相关论文对比与引用验证报告](项目相关论文对比-引用验证报告.md)
> 适用场景：`numDataSlots=10`、`numNodes=9` 的默认实验，以及后续节点数、时隙数、流量强度、拓扑密度消融。

## 一、核心结论

后续实验不宜只报告 PPO 的 `avg_r`。论文中的 MAC/TDMA 工作通常围绕吞吐、碰撞、时延、能耗、丢包、公平性、收敛速度展开；其中本项目当前已经能直接记录吞吐、请求成功率、控制冲突、队头等待、队列长度、Jain 公平性和逐时隙申请结果。

推荐主指标：

1. **归一化时隙吞吐**：每帧成功传输次数 / 数据时隙数。
2. **Tx 成功率**：成功传输次数 / slot 申请次数。
3. **冲突强度**：每帧控制冲突时隙数 `Cctrl`、每节点申请失败/碰撞时隙数 `Ncoll`。
4. **队头等待时延**：`Wt`。
5. **队列压力**：`Qt`、`sum_queue`、`traffic_rate`。
6. **公平性**：全局 `jain_tx`、局部 `Jlocal`、吞吐份额 `Sharet`。
7. **收敛速度**：达到稳定吞吐/低碰撞/公平阈值所需帧数。
8. **空间复用效率**：成功传输次数 / 数据时隙数；若允许同一时隙远距离并发，理论上可超过普通 TDMA 的 1 packet/slot 上限。

## 二、论文指标与性能情况

| 论文/协议 | 主要评价指标 | 论文中的性能结论 | 与本项目对比时应报告 |
|---|---|---|---|
| Bjorklund/Varbrand/Yuan 2003，STDMA 列生成 | 最小调度长度、最优/近最优 schedule bound、贪心算法与 bound 的差距 | 该类工作把 STDMA 调度建模为最小长度 schedule，列生成能给出很紧的最优下界，可作为启发式算法 benchmark | 用“达到同等业务负载所需帧数/时隙数”和“空间复用效率”近似比较；本项目不是离线全局优化，应强调在线、分布式、无全局冲突图 |
| Gronkvist/Nilsson/Yuan 2004，optimal spatial reuse TDMA | 吞吐量；normal TDMA、traffic-adaptive TDMA、optimal STDMA 三者对比 | 公开摘要说明 traffic adaptivity 与 spatial reuse 能带来 substantial throughput gains | 直接报告归一化时隙吞吐、相对纯 TDMA/启发式 TDMA 的提升比例 |
| Leone/Schiller 2013，自稳定 TDMA | 收敛时间、带宽利用率、带宽分配可预测性、低通信时延、动态拓扑恢复 | 重点不是吞吐峰值，而是拓扑变化/瞬时故障后的短收敛期和可证明恢复 | 增加“扰动实验”：改变拓扑或流量后，统计恢复到低冲突和高 Jain 的帧数 |
| Z-MAC 2005 | channel utilization、throughput、energy、latency、fairness；一跳/两跳/多跳拓扑 | 高竞争下吞吐可显著高于 B-MAC，公开资料报告 high contention 下超过 3 倍吞吐；低竞争下保持 CSMA 式低延迟 | 做低/中/高负载分段比较：本项目是否像 Z-MAC 一样低负载不牺牲延迟、高负载降低碰撞 |
| TRAMA / Energy-efficient collision-free MAC | 能耗、collision-free、节点睡眠、traffic-adaptive slot election | 通过分布式选举避免单播/广播碰撞，并允许非收发节点进入低功耗 idle/sleep | 本项目暂无能耗模型，先用“空闲/未申请比例”和“成功率/碰撞率”代理；若后续加能耗，按 TX/RX/idle/sleep 计费 |
| RL-MAC 2006 | throughput、power consumption、traffic adaptability、复杂度 | 使用 RL 推断其他节点状态，在多种流量下取得高吞吐和低功耗 | 对比 RL 前后：吞吐、冲突、等待、不同流量强度下稳定性；强调本项目动作粒度是逐时隙申请而非 duty cycle |
| DLMA 2018 | 归一化 throughput、个体吞吐、收敛速度；窗口平均成功时隙数 | DLMA 学习与 TDMA/q-ALOHA/FW-ALOHA/EB-ALOHA 共存，目标是最大化总吞吐；吞吐按最近 N 个时隙成功数归一化 | 采用滑动窗口吞吐曲线；报告总吞吐、每节点吞吐、公平性和达到平台期的帧数 |
| SOMAC 2020 | throughput、delay、协议选择准确率/optimality | 可按目标指标动态选择 MAC；论文报告接近 oracle，并在协议选择上优于 state of the art | 本项目不是协议选择器，但可借鉴“按目标切换”：吞吐优先、时延优先、公平优先三套 reward 消融 |
| Restless Bandits 2021 | average cost、packets dropped per slot；cost 包含 energy cost 与 holding/queue cost | Whittle index 策略在平均成本和丢包上优于 Slotted ALOHA、MWS、Lyapunov drift 等基线 | 将 `Wt/Qt` 映射为 holding cost，将申请/发送映射为 energy proxy，构造综合 cost：`C = a*Qt + b*Wt + c*req_sent - d*success` |
| MDRL-MAC 2024 | network throughput；多节点独立学习可用空闲时隙；长传播延迟下的 state/action/reward 设计 | 多智能体 DRL 用于水声认知网络中的信道和时隙共享，重点是长传播延迟下优化吞吐 | 可作为“多智能体独立学习”背景；本项目不含水声长传播延迟，避免直接数值对标 |
| RL-BMAC 2025 | throughput、packet drop rate、energy efficiency、latency、network lifetime | 公开论文报告相对 B-MAC：吞吐 +58.5%、丢包 -44.8%、能效 +35%、时延 -26.93% | 作为近期 RL-MAC 参考；本项目应补齐 packet drop/energy proxy，否则只能比较吞吐、等待和成功率 |

## 三、本项目可直接采集的指标

当前 C++ 输出三类 CSV：

| 文件 | 字段 | 对应论文指标 |
|---|---|---|
| `slot_stats.csv` 或 `slot_stats_*.csv` | `totalSlotRequests`、`totalSuccessfulTx`、`totalSuccessfulPackets`、优先级生成数 | 请求数、成功传输数、成功率、业务负载 |
| `frame_metrics.csv` 或 `frame_metrics_*.csv` | `Cctrl`、`Qt`、`Wt`、`Sharet`、`Jlocal`、`req_candidates`、`req_sent`、`req_rate_observed`、`req_prob_avg` | 控制冲突、队列、等待时延、局部公平性、申请行为 |
| `fairness.csv` 或 `fairness_*.csv` | `jain_tx`、`sum_delta_packets`、`sum_queue`、`sum_arrivals`、`sum_queue_delta`、`traffic_rate` | 全局公平性、全网吞吐、队列压力、负载 |

建议派生指标：

| 指标 | 公式 | 说明 |
|---|---|---|
| 归一化时隙吞吐 | `sum_delta_packets / numDataSlots` | 对齐 DLMA/PPOMA 的 per-slot throughput 口径 |
| Tx 成功率 | `ΔtotalSuccessfulTx / ΔtotalSlotRequests` | 判断 RL 是否只是减少申请，而非真正提升接入效率 |
| 真实业务成功率 | `ΔtotalSuccessfulPackets / ΔtotalSlotRequests` | 区分 dummy-fill/占位成功和真实业务包 |
| 控制冲突率 | `mean(Cctrl) / numDataSlots` | 对齐 collision-free MAC、Z-MAC/TRAMA 的冲突目标 |
| 平均队头等待 | `mean(Wt)` | 对齐 delay/latency 指标 |
| 平均队列长度 | `mean(Qt)` 或 `mean(sum_queue / numNodes)` | 对齐 holding cost、排队时延压力 |
| Jain 公平性 | CSV 已输出 `jain_tx` | 对齐 Z-MAC、公平调度、空间复用调度 |
| 申请保守度 | `req_sent / req_candidates` | 判断策略是否通过沉默降低碰撞 |
| 空间复用效率 | `sum_delta_packets / numDataSlots`，并结合 `Bown/T2hop` 判断并发 | 大于 1 表示一帧平均每数据时隙超过一次成功发送 |
| 收敛帧数 | 首次连续 K 帧满足阈值 | 建议 K=50 或 100，避免偶然达标 |

## 四、建议实验矩阵

### 1. 基线协议

至少保留四组：

| 组别 | 作用 |
|---|---|
| `heuristic_only` | 当前启发式 TDMA/退避策略，作为最重要本地基线 |
| `ppo_lstm` | 本项目主方案 |
| `random_or_uniform` | 判断 RL 是否显著优于随机申请 |
| `oracle/offline_upper_bound` | 可选；小规模拓扑用贪心/ILP 近似 STDMA 上界 |

如果实现成本可控，再加入：

| 组别 | 作用 |
|---|---|
| `traffic_adaptive_tdma` | 对齐 Gronkvist 2004 的 traffic-adaptive TDMA |
| `zmac_like_priority` | owner 优先、非 owner 偷空闲 slot，用来对比混合 MAC 思路 |
| `bandit_index_proxy` | 用 `Qt/Wt` 构造 index，近似 Restless Bandits 的低复杂度学习基线 |

### 2. 负载维度

| 变量 | 建议取值 | 观察重点 |
|---|---|---|
| `trafficArrivalRate` | 低、中、高、过载 | 是否在高负载下仍能降低碰撞 |
| `numNodes / numDataSlots` | 0.5、0.9、1.2、1.5 | 从轻载到过载的退化曲线 |
| burst traffic | 开/关 | RL 是否能适应突发队列 |
| adaptive/ramp traffic | 开/关 | 与项目已有动态流量控制交互是否稳定 |

### 3. 拓扑/空间复用维度

| 变量 | 建议取值 | 观察重点 |
|---|---|---|
| 平均邻居数 | 稀疏、中等、密集 | 稀疏时应出现空间复用收益，密集时应退化到近似普通 TDMA |
| 两跳冲突强度 | 低/高 | 隐藏终端场景下冲突率和吞吐 |
| 节点数 | 6、9、12、18 | 可扩展性 |
| 时隙数 | 6、10、15 | slot 资源变化下的吞吐/公平性 |

### 4. 学习维度

| 变量 | 建议取值 | 观察重点 |
|---|---|---|
| reward | `{+1,-1,0}`、吞吐+公平、吞吐+时延、综合 cost | 对齐 SOMAC 和 Restless Bandits 的多目标优化 |
| entropy coefficient | 低/中/高 | 收敛速度和公平性 |
| per-slot reward | 开/关 | 信用分配收益 |
| LSTM | 开/关或 hidden size 消融 | 历史观测是否提升稳定性 |

## 六、目标阈值建议

这些阈值不是论文中的通用定论，而是结合本项目已有记录和论文常用指标设定的实验目标。

| 指标 | 最低目标 | 理想目标 | 说明 |
|---|---:|---:|---|
| Tx 成功率 | `>70%` | `>85%` | 现有记录曾设 `>70%` 作为目标 |
| 每帧控制冲突率 | `<0.10` | `≈0` | 对齐 collision-free 方向 |
| Jain 公平性均值 | `>0.75` | `>0.85` | 避免 RL 通过牺牲部分节点吞吐换总吞吐 |
| 达标帧比例 `Jain >= 0.85` | `>30%` | `>50%` | 现有记录曾用 `>50%` 作为目标 |
| 平均队头等待 `Wt` | `<0.20s` | `<0.05s` | 现有目标为 `<0.05s`，短期可先用 `<0.20s` |
| 归一化时隙吞吐 | 高于启发式 `>15%` | 高于启发式 `>30%` | 用相对提升比直接跨实验配置更稳 |
| 收敛帧数 | `<1000` | `<300` | 对齐 DLMA/自稳定 TDMA中的 convergence 关注点 |

## 七、对比时的注意事项

1. **不要直接拿不同论文的百分比当硬基准**。Z-MAC、RL-BMAC、DLMA 的场景、物理层、负载模型和包长都不同；适合比较“指标类别”和“相对本地基线提升”，不适合比较绝对吞吐。
2. **必须同时报告吞吐和公平性**。只报告总吞吐会掩盖部分节点沉默或饥饿的问题。
3. **必须报告申请保守度**。本项目的 RL 可能通过减少申请降低碰撞，导致 `avg_r` 变好但业务吞吐未提升。
4. **空间复用要和普通 TDMA 分开解释**。普通 TDMA 的上限约是每数据时隙一次成功发送；空间复用允许远距离节点并发，因此应报告 `sum_delta_packets / numDataSlots`。
5. **收敛速度要用滑动窗口判断**。单帧 Jain 或吞吐达标不代表策略稳定。
6. **能耗暂时只能做代理指标**。当前仿真没有 TX/RX/idle/sleep 能耗模型，不能直接宣称能效提升；可以先报告 `req_sent`、空闲比例、碰撞减少作为能耗趋势代理。

## 八、当前普适性筛选对论文结论的约束

2026-04-26 已完成一轮 `N=6/9/12/16 × line/ring/star/grid/clustered/full × baseline/B × seed=1/2 × 5000s` 的广覆盖筛选，结果根目录为 `logs/generalization_20260426_111420`。96 个 run 的仿真和网络指标均完整生成，但训练步数不足以证明模型收敛；因此该轮实验只能作为“普适性筛选”，不能作为最终收敛性能表。

### 1. 当前 B 不应写成普适提升

按 24 个场景比较 B 与 baseline：

| 指标 | B 优于 baseline | B 劣于 baseline | 论文解释 |
|---|---:|---:|---|
| `packets/frame` | 11/24 | 13/24 | 吞吐优势不稳定 |
| `queue_tail100` | 10/24 | 14/24 | 队列压力优势不稳定 |
| `tx_success_rate` | 11/24 | 13/24 | 请求成功率基本持平 |
| `jain_tx_tail100` | 10/24 | 14/24 | 公平性未形成稳定优势 |

可保留的正向表述：

> B 在部分具有局部空间复用结构的拓扑中有收益，尤其是 `grid` 和 `clustered` 场景；但在中心瓶颈或强耦合拓扑中不稳定。

应避免的表述：

> B 在不同节点规模和拓扑下稳定优于 baseline。

### 2. 正例与反例应同时报告

当前筛选中较有代表性的正例：

| 场景 | 主要变化 | 可解释方向 |
|---|---|---|
| `N12_grid` | `packets/frame +0.232`，`queue_tail100 -2592.9` | 网格结构存在局部空间复用机会 |
| `N16_grid` | `packets/frame +0.078`，`queue_tail100 -818.3`，`Jain +0.0809` | 大规模网格下 B 可改善队列和公平性 |
| `N16_clustered` | `packets/frame +0.208`，`queue_tail100 -2626.7` | 聚簇结构下 B 有一定收益 |

当前筛选中较有代表性的反例：

| 场景 | 主要变化 | 风险解释 |
|---|---|---|
| `N16_ring` | `packets/frame -0.557`，`queue_tail100 +549.5`，`Jain -0.0568` | 环形高规模场景下 B 退化明显 |
| `N12_star` | 吞吐下降、队列上升、公平性下降 | 中心瓶颈拓扑中正则可能限制有效竞争 |
| `N9_full` | 吞吐下降、队列上升 | 强耦合拓扑中空间复用机会少 |

论文中应将这些反例作为边界条件，而不是剔除。

### 3. 收敛性仍需单独验证

`5000s` 的筛选实验中 PPO update 数随节点规模变化明显：

| 节点数 | PPO update 状态 | 结论 |
|---|---|---|
| N6 | 约 177 update | 可用于筛选，不足以证明收敛 |
| N9 | 约 139 update | 可用于筛选，不足以证明收敛 |
| N12 | 约 114 update | 可用于筛选，不足以证明收敛 |
| N16 | 部分 run 仅 17/27/93 update，部分 PPO 日志不完整 | 只能使用网络指标作早期行为参考 |

多数可用日志的 entropy 仍在 `0.692` 左右，接近二分类动作最大熵 `ln(2)=0.693`，说明策略大多仍处在高探索状态。后续正式论文图表应对代表场景补充长训练，至少报告：

- `avg_r` 是否进入平台期
- `entropy` 是否从 `0.693` 明显下降并稳定
- `packets/frame` 滑动窗口是否稳定
- `queue_tail100` 是否持续下降或进入平台
- `jain_tx_tail100` 是否稳定

推荐长训练复核场景：

| 类型 | 场景 |
|---|---|
| 正例复核 | `N12_grid`、`N16_grid`、`N16_clustered` |
| 反例复核 | `N16_ring`、`N12_star`、`N9_full` |

### 4. 当前论文结论建议

当前最稳妥的论文级结论是：

> 当前 B 方案在固定 9 节点长训练中具备一定网络层收益；在多节点、多拓扑广覆盖筛选中，B 的收益呈场景依赖性。它更适合存在局部空间复用机会的拓扑，对中心瓶颈和强耦合拓扑不稳定。后续需要拓扑自适应正则和 reward 对齐，才能形成普适算法结论。

## 九、建议补充的说明性性能参数

本节基于已核验论文口径补充更有解释力的指标。`academic-refchecker` 对 Z-MAC、DLMA、SOMAC、Restless Bandits、optimal spatial reuse TDMA 等核心论文均能匹配到真实论文；TRAMA、DLMA、Restless Bandits、自稳定 TDMA 存在会议/期刊或 arXiv 年份差异，正式引用时按 DOI/出版源修正。RL-MAC 被 refchecker 报作者错误，但项目已有验证报告确认这是 PDF 作者行抽取问题，不能据此判定论文虚构。

### 1. 论文主结果表建议使用的一级指标

| 指标 | 建议公式 | 说明性价值 | 参考论文口径 |
|---|---|---|---|
| 有效负载吞吐 `goodput_per_slot` | `ΔtotalSuccessfulPackets / (frames * numDataSlots)` | 比单纯 successful tx 更能说明真实业务包交付能力，避免 dummy/占位成功误导 | DLMA、Z-MAC、TRAMA 均以吞吐/交付能力为核心 |
| 数据包交付率 `packet_delivery_ratio` | `ΔtotalSuccessfulPackets / Δarrivals`，若 arrivals 不完整则用生成包数近似 | 说明网络是否真的消化业务负载，而不是只减少申请 | Z-MAC、TRAMA、RL-BMAC 常报告 delivery/drop 类指标 |
| 丢包/积压率 `drop_or_backlog_rate` | `1 - packet_delivery_ratio` 或 `Δsum_queue / Δarrivals` | 当前系统可能不显式丢包，队列增长可作为未服务业务的等价风险 | Restless Bandits 使用 packets dropped per slot 和 average cost |
| MAC 接入效率 `mac_efficiency` | `ΔtotalSuccessfulPackets / ΔtotalSlotRequests` | 区分“积极申请但低效”和“少申请但高效” | collision-free MAC、TRAMA、Z-MAC |
| 空间复用增益 `spatial_reuse_gain` | `goodput_per_slot / tdma_goodput_per_slot` | 直接回答空间复用是否超过普通 TDMA/启发式 TDMA | optimal spatial reuse TDMA、STDMA 调度 |
| 可并发度 `concurrent_success_degree` | `sum_delta_packets / active_data_slots` | 说明同一帧/同一调度周期内平均有多少有效并发传输 | STDMA/空间复用调度 |
| 队列稳定性 `queue_stability_slope` | 对 `sum_queue` 做末段线性拟合斜率 | 判断系统是否处于稳定服务区；比单点 `queue_tail100` 更可靠 | Restless Bandits 的 holding cost/queue cost 思路 |
| P95 队头等待 `Wt_p95` | `Wt` 的 95 分位数 | 平均等待可能掩盖尾部拥塞；P95 更适合说明 QoS | SOMAC 支持 delay 作为优化目标，自稳定 TDMA关注低延迟 |
| 公平吞吐 `fair_goodput` | `goodput_per_slot * jain_tx` | 将吞吐和公平性合成，避免高吞吐来自少数节点独占 | DLMA 的 alpha-fairness、Z-MAC fairness |
| 饥饿节点比例 `starvation_ratio` | 末段窗口内 `delta_packets == 0` 的节点数 / `numNodes` | 比 Jain 更直观地说明是否有节点长期拿不到服务 | fairness/throughput tradeoff 类 MAC 评价 |
| 控制面开销 `control_pressure` | `Cctrl / numDataSlots`，可加 `req_sent / numDataSlots` | 说明吞吐提升是否靠大量控制竞争换来 | TRAMA/Z-MAC 都关注碰撞和调度开销 |
| 能耗代理 `energy_proxy` | `a*req_sent + b*Cctrl + c*active_slots` | 当前无真实能耗模型时，先用申请、冲突、活跃时隙近似 | TRAMA、Z-MAC、RL-BMAC 均报告 energy/power |

### 2. 推荐的论文主表结构

主表不建议继续以 `avg_r` 为第一列。`avg_r` 应放在训练诊断表中，论文性能主表建议如下：

| 场景 | 方法 | goodput/slot | spatial reuse gain | PDR | MAC efficiency | queue slope | Wt P95 | Jain | starvation | control pressure |
|---|---|---:|---:|---:|---:|---:|---:|---:|---:|---:|
| N9 默认 | baseline |  |  |  |  |  |  |  |  |  |
| N9 默认 | B |  |  |  |  |  |  |  |  |  |

这张表能同时回答四个关键问题：

1. **吞吐是否提升**：看 `goodput/slot` 和 `spatial reuse gain`。
2. **提升是否真实服务了业务**：看 `PDR`、`queue slope`、`Wt P95`。
3. **是否牺牲部分节点**：看 `Jain` 和 `starvation`。
4. **是否靠过度控制竞争换性能**：看 `MAC efficiency` 和 `control pressure`。

### 4. 建议优先级

第一优先级，适合马上加入汇总脚本和论文表格：

- `goodput_per_slot`
- `packet_delivery_ratio`
- `mac_efficiency`
- `queue_stability_slope`
- `Wt_p95`
- `fair_goodput`
- `starvation_ratio`
- `control_pressure`

第二优先级，适合作为扩展实验或消融：

- `spatial_reuse_gain`
- `concurrent_success_degree`
- `energy_proxy`
- `convergence_to_goodput_threshold`

### 5. 写论文时的解释口径

如果 B 的 `avg_r` 不如 baseline，但 `goodput_per_slot` 更高、`queue_stability_slope` 更低、`starvation_ratio` 不升高，可以解释为：

> PPO reward 与网络层目标不完全一致；B 牺牲了部分逐时隙即时奖励，但提升了长期业务服务能力。

如果 B 的 `goodput_per_slot` 更高但 `starvation_ratio` 也升高，应解释为：

> B 存在吞吐-公平性折中，不能单独宣称整体性能提升，需要配合公平正则或拓扑自适应机制。

如果 B 的 `mac_efficiency` 提高但 `goodput_per_slot` 不提高，应解释为：

> B 主要学会减少无效申请，表现为避撞保守，而不是真实吞吐提升。

## 十一、相关论文年份、影响因子与影响力分层

影响因子只适用于期刊；INFOCOM、VTC、SenSys、ICC、ICNSC 等会议没有 JCR Impact Factor。对会议论文应看会议声誉、引用量和领域认可度，不能用 IF 直接排序。

| 论文/协议 | 年份 | Venue | 类型 | 近期 JCR IF/说明 | 影响力判断 | 本项目用途 |
|---|---:|---|---|---:|---|---|
| Z-MAC | 2005 / 2008 | SenSys 2005；IEEE/ACM Transactions on Networking 2008 | 会议 + 期刊 | TON 2024 IF 约 3.6 | 高 | 混合 CSMA/TDMA、高竞争吞吐和公平性对比 |
| TRAMA / Energy-efficient collision-free MAC | 2003 / 2006 | SenSys 2003；Wireless Networks 2006 | 会议 + 期刊 | Wireless Networks 2024 IF 约 2.1 | 高 | traffic-adaptive、collision-free、能耗/睡眠基线 |
| Resource optimization of spatial TDMA | 2003 | IEEE INFOCOM 2003 | 会议 | 无 IF | 高 | STDMA 离线优化和调度上界 |
| Throughput of optimal spatial reuse TDMA | 2004 | IEEE VTC 2004-Spring | 会议 | 无 IF | 中高 | normal TDMA、traffic-adaptive TDMA、optimal STDMA 对比 |
| DLMA | 2018 | IEEE ICC 2018 | 会议 | 无 IF | 高 | DRL 多址/时隙接入代表工作 |
| RL-MAC | 2006 | IEEE ICNSC 2006 | 会议 | 无 IF | 中 | 早期 RL-MAC 背景 |
| SOMAC / automatic MAC protocol selection | 2020 | Computer Communications | 期刊 | 2024 IF 约 4.3 | 中高 | RL 协议选择、按目标指标自适应 |
| Restless Bandits spatial reuse scheduling | 2021 | Performance Evaluation | 期刊 | 当前 IF 约 0.8 | 中 | 非深度 RL/index policy 空间复用基线 |
| Self-Stabilizing TDMA | 2013 | International Journal of Distributed Sensor Networks | 期刊 | 2024 IF 约 2.5 | 中 | 动态拓扑恢复与收敛性对比 |
| RL-MAC survey | 2023 | Entropy | 期刊 | 2024 IF 2.0 | 中 | related work 分类框架 |
| MDRL-MAC underwater multiple access | 2024 | Computers and Electrical Engineering | 期刊 | 2024 IF 约 4.9 | 中 | 多智能体 DRL MAC 背景，避免直接数值对标 |
| RL-BMAC | 2025 | Information | 期刊 | 2024 IF 2.9 | 中 | 近期 RL-MAC 指标口径参考 |

影响力分层建议：

1. **核心相关工作**：Z-MAC、TRAMA、STDMA/optimal spatial reuse TDMA、DLMA。这些文献最适合放在 related work 的主线中。
2. **方法或指标参考**：SOMAC、Restless Bandits、Self-Stabilizing TDMA、RL-MAC。它们用于补充“协议选择、index policy、收敛恢复、早期 RL-MAC”。
3. **近期背景**：RL-BMAC、MDRL-MAC、Entropy survey。它们适合说明近期 RL-MAC 方向和性能指标，但不应作为最权威 benchmark。

和本项目当前结果的关系：

- `N9_ring` 中 B 的 `goodput/slot` 相对 baseline 提升约 62.5%，与 RL-BMAC 报告的吞吐提升 58.5% 属于同量级。
- Z-MAC 文献中高竞争场景相对 B-MAC 的吞吐提升常见为 20-40% 量级；本项目在 `N9_ring` 的相对提升更高，但公平性略降，且没有真实能耗模型。
- TRAMA/Z-MAC 同时强调 collision-free、energy 或 fairness；本项目当前只能报告 control pressure、energy proxy 和 Jain，不能宣称真实能效优势。
- Optimal STDMA 文献关注上界；本项目尚未引入 oracle/ILP/greedy STDMA，因此不能宣称接近最优空间复用 TDMA。

论文表述建议：

> 在 `N9_ring` 场景中，B 相对本地 baseline 获得约 62.5% 的 goodput/PDR 提升，达到若干 RL/混合 MAC 文献中相对基线改进的同量级；但公平性略有下降，且尚未对比 optimal STDMA 上界，因此应表述为“相对本地 baseline 的显著网络层收益”，而不是“超过经典文献”。

## 十二、统一 benchmark 框架入口

为避免后续“文献对比”散落在不同脚本和目录中，新增统一入口 `scripts/benchmark_suite.sh`。该入口当前不绑定具体文献模型，只负责把场景、方法、seed、日志、指标和汇总结果组织成固定结构。

### 1. 目录规范

```text
logs/benchmark_<suite>_<YYYYMMDD_HHMMSS>/
  manifest.tsv
  suite_config.tsv
  N9_ring/
    baseline/
      seed1/
      seed2/
    B/
      seed1/
      seed2/
  summaries/
    metrics_summary.csv
    metrics_method_summary.csv
    metrics_scenario_method_summary.csv
    comparison_table.md
```

其中 `scenario/method/seed` 是最小结果隔离单元；`manifest.tsv` 记录计划运行项；`suite_config.tsv` 记录本轮 benchmark 的统一参数；`summaries/` 只放汇总产物，便于论文表格引用。

### 2. 方法注册

方法统一在 `configs/benchmark_methods.tsv` 注册。当前第一版只注册已有方法：

注册表字段固定为：

```text
method  implementation  network  runner  macMode  adapter  needs_rl  description
```

字段含义：

| 字段 | 含义 |
|---|---|
| `method` | benchmark 层的对比方法名，例如 `baseline`、`B`、`zmac_like` |
| `implementation` | 实际 OMNeT++ 实现，例如 `DynamicTDMA` 或未来独立协议模块 |
| `network` | NED network 名称，例如 `TDMANetwork` |
| `runner` | 启动方式，例如 `ablation`、`omnet_only`、`omnet_custom`、`external` |
| `macMode` | 同一 implementation 内部的协议模式 |
| `adapter` | 指标归一化方式，例如 `dynamic_tdma` 或 `canonical` |
| `needs_rl` | 是否需要启动 PPO/RL 训练器 |
| `description` | 方法说明 |

当前映射：

| method | implementation | network | runner | macMode | adapter | needs_rl | 说明 |
|---|---|---|---|---|---|---|---|
| `baseline` | `DynamicTDMA` | `TDMANetwork` | `ablation` | `dynamic_tdma` | `dynamic_tdma` | true | 当前本地基线 |
| `B` | `DynamicTDMA` | `TDMANetwork` | `ablation` | `dynamic_tdma` | `dynamic_tdma` | true | 当前主线方案 |
| `D` | `DynamicTDMA` | `TDMANetwork` | `ablation` | `dynamic_tdma` | `dynamic_tdma` | true | 邻居密度自适应消融 |
| `DB` | `DynamicTDMA` | `TDMANetwork` | `ablation` | `dynamic_tdma` | `dynamic_tdma` | true | D+B 组合消融 |
| `external` | `external` | `-` | `external` | `-` | `canonical` | false | 预留给手工导入外部论文结果 |

后续如果实现 `zmac_like`、`trama_like`、`greedy_stdma` 或 oracle 上界，应作为新的 `method` 注册，而不是改动目录结构。

示例：

### 4. 论文表述边界

当前框架只保证“同场景、同 seed、同指标、同目录结构”。它还没有实现 Z-MAC、TRAMA、STDMA oracle 等具体文献协议，因此不能写成“已经完成与 Z-MAC/TRAMA 的同场景公平复现对比”。

正确表述：

> 本文建立了统一 benchmark 框架，可在相同拓扑、流量、随机种子和指标口径下比较不同 MAC/学习方法；当前实验先比较本地 baseline 与 B，后续文献协议可作为独立 method 接入。

错误表述：

> 当前结果已经证明 B 优于 Z-MAC 或 TRAMA。

## 十三、跨协议统一实现规范

如果某个 TDMA/MAC 文献协议的时隙结构、控制阶段或 superframe 设计和当前 DynamicTDMA 不同，仍可进入统一 benchmark，但必须区分以下层级：

```text
platform = OMNeT++
implementation = DynamicTDMA.cc / future protocol cc
macMode = 同一 implementation 内的协议模式
method = benchmark 对比方法名
runner = 启动方式
adapter = 指标归一化方式
```

建议第一阶段继续把 `DynamicTDMA.cc` 作为统一 OMNeT++ 入口：共享拓扑、流量、队列、日志、metrics 和 benchmark 目录；协议差异通过 `macMode` 切换。只有当论文协议的状态机、控制帧流程、sleep/listen 周期或 superframe 结构差异很大时，才新建独立 `.cc/.ned`。

### 1. 统一 `.cc` 入口的边界

适合放进 `DynamicTDMA.cc + macMode` 的情况：

- 只改变时隙选择、申请概率、优先级或退避策略。
- 只新增 owner slot、slot stealing、traffic-adaptive election 等轻量规则。
- 仍然复用当前拓扑、流量、队列和主要 metrics。

应考虑新建独立实现的情况：

- 控制帧阶段完全不同。
- superframe 结构完全不同。
- 协议需要 sleep/listen 周期或独立能耗状态机。
- 节点状态机与当前 RTS/CTS/data 流程不兼容。

### 2. Canonical metrics

跨协议主表优先使用 canonical metrics，而不是只看 TDMA 内部指标：

```text
goodput_per_second
packet_delivery_ratio
p95_delay_s
jain_fairness
control_overhead_ratio
energy_proxy
```

`goodput_per_slot`、`packets/frame`、`slot_util_mean`、`spatial_reuse_gain` 仍可保留，但它们只适合同类 TDMA/STDMA 结构内部解释。不同时隙结构之间比较时，必须优先看按仿真时间和业务交付归一化的指标。

### 3. 实现规范 Skill

新增 `.codex/skills/implement-mac-protocol/SKILL.md`，用于后续实现或规划新 MAC/TDMA/论文协议时自动套用以下流程：

1. 判断协议应放入 `DynamicTDMA.cc` 的 `macMode`，还是新建独立 `.cc/.ned`。
2. 注册 benchmark method：`implementation/network/runner/macMode/adapter/needs_rl`。
3. 隔离共享仿真层和协议差异层。
4. 输出或转换 canonical metrics。
5. 更新 benchmark 文档和算法记录。
6. 运行 smoke test 后再进入正式实验。

- Gronkvist/Nilsson/Yuan optimal spatial reuse TDMA：公开摘要说明比较 normal TDMA、traffic-adaptive TDMA、optimal STDMA，并指出 traffic adaptivity 与 spatial reuse 带来显著吞吐收益。<https://www.researchgate.net/publication/4121056_Throughput_of_optimal_spatial_reuse_TDMA_for_wireless_ad-hoc_networks>
- Leone/Schiller 自稳定 TDMA：SAGE 页面摘要和正文说明关注 high bandwidth utilization、predictability、low delay 和短收敛期。<https://journals.sagepub.com/doi/10.1155/2013/639761>
- Z-MAC：公开资料说明其在低竞争下类似 CSMA、高竞争下类似 TDMA，并报告高竞争尤其两跳场景下吞吐超过 B-MAC 三倍。<https://www.sciweavers.org/publications/z-mac-hybrid-mac-wireless-sensor-networks>
- TRAMA：公开摘要说明其 traffic-adaptive、collision-free，并通过非收发节点 idle/sleep 降低能耗。<https://www.researchgate.net/publication/2933600_Energy-Efficient_Collision-Free_Medium_Access_Control_for_Wireless_Sensor_Networks>
- DLMA：公开页面说明吞吐按最近窗口成功时隙数归一化，评估与 TDMA/q-ALOHA 等共存。<https://www.researchgate.net/publication/321487567_Deep-Reinforcement_Learning_Multiple_Access_for_Heterogeneous_Wireless_Networks>
- SOMAC：ScienceDirect 页面说明支持以 throughput、delay、packet drop rate 等数值反馈为优化目标，并报告协议选择接近 oracle。<https://www.sciencedirect.com/science/article/pii/S0140366419311284>
- Restless Bandits spatial reuse scheduling：ScienceDirect 页面说明指标为 average cost 和 packets dropped per slot，目标是最小化能耗和传输时延。<https://www.sciencedirect.com/science/article/pii/S0166531621000250>
- MDRL-MAC 水声多址：ScienceDirect 页面说明多智能体 DRL 学习可用空闲时隙并优化 network throughput。<https://www.sciencedirect.com/science/article/abs/pii/S0045790624007468>
- RL-BMAC：MDPI 全文报告 throughput、packet drop rate、network lifetime、energy consumption、latency，并给出相对 B-MAC 的提升比例。<https://www.mdpi.com/2078-2489/16/5/369>

## 十五、后续仿真条件补齐规范

当前已完成的正式结果主要证明：在统一 PPO 训练口径下，B 相对 PPO baseline 在部分拓扑场景有效。为了支撑“PPO-TDMA 相对传统 TDMA、启发式 TDMA，在动态 MANET 背景下的优势”，后续 benchmark 必须补齐传统基线、动态负载、拓扑扰动、节点上下线和多业务 QoS 条件。研究背景与仿真条件的对应关系见 [研究进度与背景.md](研究进度与背景.md)。

### 1. 方法层级

后续主表不再只使用 `baseline/B`，而应扩展为：

```text
plain_tdma vs heuristic_only vs ppo_baseline vs B
```

含义如下：

| method | runner | macMode | needs_rl | 含义 |
| :--- | :--- | :--- | :--- | :--- |
| `plain_tdma` | `omnet_only` | `plain_tdma` | false | 固定 owner slot 的传统 TDMA，无 PPO、无学习 |
| `heuristic_only` | `omnet_only` | `heuristic_only` | false | 当前启发式动态申请规则，不启动 PPO |
| `ppo_baseline` | `ablation` | `dynamic_tdma` | true | 当前 PPO baseline 的显式别名 |
| `B` | `ablation` | `dynamic_tdma` | true | PPO baseline + heuristic deviation regularization |

`plain_tdma` 和 `heuristic_only` 必须作为非学习基线单独报告，不能再把 `ppo_baseline` 称为传统 TDMA。

### 2. 负载条件

负载扫描统一使用场景后缀表达：

| 后缀 | 配置含义 |
| :--- | :--- |
| `_load_low` | `trafficArrivalRate=1.0` |
| `_load_medium` | `trafficArrivalRate=5.0` |
| `_load_high` | `trafficArrivalRate=10.0` |
| `_load_overload` | `trafficArrivalRate=20.0` |
| `_ramp` | `enableRampTraffic=true` |
| `_adaptive` | `enableAdaptiveTraffic=true` |

示例：

```text
N9_ring_load_low
N9_ring_load_high
N12_grid_ramp
```

负载扫描重点看 `goodput_per_second`、`packet_delivery_ratio`、`queue_stability_slope`、`Wt_p95`、`jain_fairness`。如果 reward 与网络指标冲突，以网络指标解释为准。

### 3. 动态拓扑和节点变化

第一阶段采用离散扰动，不直接实现连续坐标移动，也不运行时增删 NED submodule：

```text
static
edge_toggle
topology_switch
node_dropout
node_rejoin
```

节点变化优先用 active/inactive 状态模拟：inactive 节点停止生成业务、停止申请时隙，必要时停止转发。这样可以保持 `numNodes` 和状态向量维度稳定，避免 PPO 输入维度在运行中变化。

推荐场景：

```text
N9_ring_to_grid
N12_grid_edge_toggle
N12_star_center_stress
N16_clustered_bridge_break
N12_grid_node_dropout
N12_grid_node_rejoin
```

动态场景需要额外记录扰动发生帧，并在汇总中区分扰动前、扰动后和恢复窗口。重点指标为恢复时间、PDR 下降幅度、队列峰值和 Jain 恢复速度。

### 4. 多业务 QoS

后续业务生成至少区分：

```text
high_priority: 低时延、高可靠
best_effort: 普通数据
```

QoS 指标应新增：

```text
high_priority_pdr
high_priority_p95_delay
best_effort_pdr
priority_fairness
```

这部分用于支撑研究背景中的“多业务服务质量差异化”，但不能只看总吞吐；如果高优先级可靠性提升是以 best-effort 完全饥饿为代价，需要在结论中明确说明。
