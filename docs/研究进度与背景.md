# 研究进度与背景

> 本文档汇总 DynamicTDMA 项目的研究背景、系统架构、已完成工作和关键实验结果。详细的算法演化记录见 [算法改进记录.md](算法改进记录.md)。

## 一、研究背景

### 1.1 问题定义

在多跳无线自组织网络（MANET）中，**时分多址（TDMA）** 需要在有限的时隙资源上协调多个节点的传输。核心挑战：

- **碰撞规避**：多个节点同时申请同一时隙会导致传输失败
- **空间复用**：非邻居节点可在同一时隙传输而不互相干扰
- **公平性**：避免某些节点长期饿死
- **动态负载**：业务强度随时间变化，固定分配策略无法适应

### 1.2 研究目标

设计一个基于**在线强化学习**的动态 TDMA MAC 协议，让每个节点通过局部特征自主学习最优申请策略，在保证公平性的前提下最大化网络吞吐、最小化碰撞。

### 1.3 约束条件

1. **完全分布式**：每节点只能使用局部观测（队列长度、邻居状态、历史碰撞等），不共享参数
2. **在线学习**：边仿真边训练，不能依赖离线数据集
3. **计算受限**：每节点的网络推理需在时隙周期内完成
4. **与现有 MAC 机制兼容**：保留 RTS/CTS 三阶段握手和空间复用机制

### 1.4 后续仿真条件与研究背景对齐

为了让实验条件支撑“高动态 MANET、节点变化、多业务 QoS、传统 TDMA 对照”的研究背景，后续 benchmark 不应只比较 `B` 与 PPO baseline，而应按统一场景补齐以下对照维度：

| 研究背景需求 | 对应仿真条件 | 主要验证目标 |
|:---|:---|:---|
| 高动态 MANET | `edge_toggle`、`topology_switch`、`node_dropout`、`node_rejoin` 等离散拓扑扰动 | 扰动后吞吐恢复、PDR 下降幅度、队列峰值、Jain 恢复速度 |
| 多节点规模变化 | `N6/N9/N12/N16` 等规模扫描，结合 `line/ring/star/grid/clustered/full` 拓扑 | 验证 PPO-TDMA 是否只适用于 9 节点单一拓扑，还是能跨规模保持收益 |
| 多业务 QoS | `high_priority` 与 `best_effort` 两类业务流 | 比较高优先级 PDR、P95 时延、普通业务可用性和优先级公平性 |
| 传统 TDMA 对照 | `plain_tdma / heuristic_only / ppo_baseline / B` | 区分传统固定时隙、非学习启发式、PPO baseline 和 B 的真实贡献 |

当前已实现可运行入口的是传统基线与负载条件：`plain_tdma` 表示固定 owner slot 的传统 TDMA，`heuristic_only` 表示当前启发式调度但不启动 PPO，`ppo_baseline` 表示显式 PPO baseline，`B` 表示加入 heuristic deviation regularization 的主线候选。动态拓扑、节点上下线和 QoS 细粒度指标仍属于下一阶段实现内容。

## 二、系统架构

### 2.1 整体组成

```
┌─────────────────────┐       命名管道        ┌─────────────────────┐
│  OMNeT++ C++ 仿真   │ ◄───────────────────► │   Python RL 训练器   │
│  (DynamicTDMA)      │    /tmp/tdma_rl_*     │   (PPO + LSTM)       │
│                     │                       │                      │
│ • 物理层建模         │   状态管道 →           │ • 特征提取            │
│ • RTS/CTS/DATA      │   动作管道 ←           │ • 策略网络推理         │
│ • 启发式调度器       │                       │ • PPO 训练循环         │
│ • 节点拓扑管理       │                       │ • 检查点保存           │
└─────────────────────┘                       └─────────────────────┘
```

### 2.2 TDMA 帧结构

每帧按顺序执行三阶段：

1. **请求阶段（Request Phase）**
   - 每节点在专属微时隙广播 `TDMAGrantRequest`（RTS）
   - 携带申请概率向量 `myPriorities[slot]`
   - 概率来自 RL 输出或启发式（乘数模式融合）

2. **回复阶段（Reply Phase）**
   - 节点对收到的 RTS 做授权决策（+1 同意 / -2 拒绝）
   - 发送 `TDMAGrantReply`（CTS）
   - `finalizeOccupancyFromCts()` 汇总所有 CTS 后构建占用表

3. **数据阶段（Data Phase）**
   - 持有 `mySlots[slot]=true` 的节点发送 `TDMADataPacket`
   - 阶段结束推送 RL 状态、读取下一帧动作

### 2.3 网络拓扑

**当前为手动配置的树形局部连接**（非全连接），9 节点：
```
   0 ─── 8 ─── 6 ─── 7
                │
                1 ─── 2
                │
                3 ─── 4
                │
                5
```

**空间复用**：节点连接关系决定一跳/两跳邻居集合，非邻居可在同一时隙并发传输。

### 2.4 RL 状态特征（每节点 69 维）

| 组 | 维度 | 含义 |
|:---|:---|:---|
| `Bown` | 10 (M) | 本节点各时隙占用位图 |
| `T2hop` | 20 (2M) | 两跳邻居占用状态 + 最小跳数 |
| 标量特征 | 10 | 队列长度、到达率、公平指数、碰撞历史等 |
| `Pt1` | 10 (M) | 上一帧申请概率向量 |
| `HeurProb` | 10 (M) | 本帧启发式基准概率（乘数模式参考） |
| `NodeId` | 9 (N) | 节点 ID one-hot |

### 2.5 网络结构（LSTM Actor-Critic，~13.8K 参数/节点）

```
输入 (B, T, 69)
    → LSTM-1 编码器 (hidden=32)
    → Actor: Linear(10) → Sigmoid → α ∈ (0,1)^10
    → Critic: Linear(10) → V_s(t) （逐时隙价值估计）
```

- **乘数模式**：RL 输出 α，C++ 计算 `reqProb = clamp(heurProb × 2α, 0, 1)`
- **二值动作**：Bernoulli 采样得到 0/1，乘数实际取 {0, 2}
- **Actor 零初始化**：Sigmoid(0)=0.5，起点等效纯启发式

## 三、关键机制

### 3.1 RL 乘数模式（核心设计）

**动机**：直接替换启发式会破坏 `avoidSlotsNextSchedule` 退避机制，历史实验验证 avg_r 从 +44 跌至 +13。

**机制**：C++ 始终计算启发式概率 `heurProb`，RL 仅作为**乘数**调整：
- α > 0.5 → 乘数 > 1 → 放大申请（激进）
- α < 0.5 → 乘数 < 1 → 抑制申请（保守）
- α = 0（动作=0）→ 乘数 = 0 → **完全不申请**（碰撞规避）

### 3.2 逐时隙奖励分解

**问题**：旧版聚合奖励 `r = Nsucc - Ncoll` 无法为 10 个独立 Bernoulli 动作分配信用——10 个动作共享 1 个标量，PPO 无法区分哪个时隙好/坏。

**改进**：C++ 推送逐时隙结果 `SlotResult`（'0' 未申请 / '1' 成功 / '2' 失败），Python 计算逐时隙奖励 `r_s ∈ {+1, -1, 0}`。PPO 改为逐时隙 `ratio × advantage`，Critic 输出 M 个逐时隙价值估计。

**效果**：entropy 收敛速度提升 **105 倍**（83 次更新下降 0.168 vs 旧版 0.0016）。

### 3.3 奖励延迟对齐

**问题**：帧 t 的 `SlotResult` 反映的是帧 t-1 动作的结果，若直接配对会导致因果时序错位。

**机制**：`pending_transitions` 将帧 t 的奖励与帧 t-1 的动作配对后再存入 Buffer。

### 3.4 EWMA 奖励基线

每节点维护逐时隙 EWMA 基线（衰减 0.95），存入 Buffer 的奖励为差分 `r_shaped = r_t - baseline`，减小方差、突出改进信号。

### 3.5 自适应 ent_coef

`ent_coef` 根据当前 entropy 自适应调整：
- entropy < 0.45 → 切换到 `ent_coef_high`（0.10）防止策略崩溃
- entropy > 0.55 → 使用基础 `ent_coef`（0.05）
- 中间区域线性插值

### 3.6 独立 Agent 架构

**每节点维护独立**的网络实例、Adam 优化器和 ExponentialLR 调度器。各节点的 `RolloutBuffer` 也独立存储，PPO 更新时逐节点独立计算梯度——完全分布式，无参数共享。

### 4.2 失败实验归档

为避免重复走弯路，以下方向已被实验否决：

- **BC 预训练**（2026-03-28）：乘数模式下 BC 目标放大启发式概率，破坏退避机制
- **soft 二值乘数 {0.5, 1.5}**（2026-04-11）：L_actor 死寂，无法学习
- **Squashed Gaussian 连续动作**（2026-04-15）：entropy 上漂，梯度过度扩散
- **-0.2 未申请惩罚**（2026-04-16）：引入噪声削弱 PPO 信号
- **ent_coef 降至 0.01**（2026-04-17）：短期边际优势，长期趋同

## 五、最新实验结果（2026-04-19）

### 5.1 30000s 长训练

- 总更新次数：**837 次**
- 初期 avg_r：-8.667 → 末期 avg_r ≈ **-0.9**
- Entropy：0.0399 → 0.0074（策略高度确定）
- 最佳单次：**avg_r = +0.000**（update 804，首次触及非负）

### 5.2 纯启发式基线（对照组）

Python 监听-only 模式，C++ 回退纯启发式：

| 策略 | avg_r 均值 | avg_r 范围 |
|:---|:---|:---|
| 纯启发式 | **-306.035** | [-362, -247] |
| RL 训练末期 | **≈ -0.9** | [-10.3, 0.0] |

**RL 相对启发式提升约 340 倍**。

### 5.3 结果解读

- **纯启发式**：每帧每节点平均 -2.4 净奖励（每帧 2.4 次碰撞）
- **RL**：每帧每节点平均 -0.007 净奖励（碰撞近乎消除）
- **`avg_r ≈ -1` 的真正含义**：接近理论最优，而非策略失败
  - 要么成功传输（+1）
  - 要么智能选择不申请（0）
  - 碰撞（-1）极少

## 六、已识别的关键约束与风险

1. **on-policy 偏差**：C++ 仿真速度快于 Python，`update_every=128` 在样本量与 off-policy 程度间取平衡。
2. **L_critic 归一化**：Critic 损失已归一化，正常量级 1~10。若出现 300+ 说明奖励未归一化或网络发散。
3. **entropy 收敛陷阱**：过早降至 < 0.01 可能陷入次优确定性策略，自适应 ent_coef 用于对冲。
4. **网络拓扑固定**：当前 9 节点树形拓扑，扩展到动态拓扑需重新验证状态向量维度。
5. **奖励函数边界**：{+1, -1, 0} 奖励已接近饱和，进一步提升空间有限。

## 七、后续研究方向

### 7.1 短期（功能验证）
- **吞吐量导向奖励**：将奖励从 {+1, -1, 0} 改为基于数据包传输成功率或延迟的指标
- **动态业务场景**：验证 RL 在变化负载（`enableAdaptiveTraffic`）下的泛化能力
- **更大规模拓扑**：扩展到 20~50 节点验证可扩展性
- **传统基线对照**：使用统一 benchmark 比较 `plain_tdma / heuristic_only / ppo_baseline / B`
- **动态条件补齐**：先用离散拓扑扰动和 active/inactive 节点模拟高动态 MANET，再扩展到 QoS 分类业务

### 7.2 中期（算法改进）
- **注意力机制替代 LSTM**：捕获邻居间依赖关系
- **分层策略**：粗粒度（申请/不申请）+ 细粒度（优先级）
- **课程学习**：从简单拓扑逐步过渡到复杂拓扑

### 7.3 长期（系统集成）
- **实机部署可行性**：研究推理延迟、模型压缩
- **多智能体博弈建模**：将问题形式化为 Dec-POMDP
- **与传统 MAC 协议对比基线**：CSMA/CA、S-TDMA 等
